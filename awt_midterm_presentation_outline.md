# AWT 중간 점검 발표 구성안

> 목적: 현재 브랜치의 진행 상황을 간단하고 명료하게 공유하기 위한 PPT 초안

---

## 1. 발표 제목

**AWT 중간 점검**

AI 기반 웹 SW 시험인증 자동화 도구 개발 현황

---

## 2. 한 줄 요약

AWT는 매뉴얼과 URL을 입력받아 **테스트케이스 설계, 검토, 자동 실행, 결과 판정**까지 지원하는 웹 SW 시험 자동화 도구이다.

현재는 PoC 검증을 마치고, 실제 오픈소스 게시판 대상 실전 실행 단계로 넘어가는 중이다.

---

## 3. 왜 필요한가

기존 시험 업무는 다음 부담이 크다.

- 매뉴얼을 읽고 테스트케이스를 직접 작성해야 한다.
- 반복 검토와 실행에 시간이 많이 든다.
- 결함 원인과 재현 정보를 정리하는 데 추가 노력이 필요하다.

AWT의 목표는 이 과정을 자동화하되, **사람 검토 단계는 유지**하는 것이다.

---

## 4. 전체 흐름

AWT는 7단계 흐름으로 동작한다.

1. URL 또는 매뉴얼 입력
2. 기능 목록 추출
3. AI가 테스트케이스 생성
4. 검증 규칙으로 품질 확인
5. 시험원이 승인 또는 수정
6. 브라우저 자동 실행
7. 결과와 결함 원인 정리

발표 화면에는 이 흐름을 단순한 화살표 다이어그램으로 넣으면 좋다.

---

## 5. 현재까지 완료한 것

현재 브랜치에서 완료된 주요 작업은 다음과 같다.

- 기본 설계 문서 정리
- PoC alpha, beta, gamma 검증 완료
- Windows 데스크탑 앱 구조 구현
- LLM Provider 구조 도입
- 테스트케이스 품질 검증 규칙 강화
- 자산 저장소 기반 개선 구조 추가

핵심은 **아이디어 단계가 아니라 실제 코드와 검증 결과가 있는 상태**라는 점이다.

---

## 6. PoC 결과

PoC에서는 간단한 게시판 예제로 전체 흐름을 검증했다.

| 항목 | 결과 |
|---|---|
| 생성 TC 수 | 41개 |
| 시험원 검토 | 41개 승인 |
| 자동 실행 | 41개 모두 실행 |
| 실행 결과 | PASS 32개, FAIL 9개 |
| 결함 검출 | 의도 결함 3종 + 신규 결함 2종 |

PoC 결과, AWT가 만든 테스트케이스가 실제 결함 검출까지 이어질 수 있음을 확인했다.

---

## 7. 발견한 결함 예시

PoC 실행 중 다음 결함을 검출했다.

- 비로그인 또는 타인 글 수정 가능
- 비로그인 또는 타인 글 삭제 가능
- 제목 길이 제한 누락
- 빈 비밀번호 로그인 처리 문제
- 글 수정 시 빈 제목 저장 가능

특히 권한 검증 누락은 실제 서비스에서도 중요한 결함 유형이다.

---

## 8. 품질 개선 내용

단순히 AI가 테스트케이스를 만들게 하는 것에 그치지 않고, 품질 검증 장치를 추가했다.

- 필수 컬럼 누락 확인
- 매뉴얼 근거 확인
- 추론 기반 테스트케이스 비율 확인
- 테스트 기법 분포 확인
- 누락 기능 확인
- 선택자 안정성 점수 확인
- 음성 테스트 카테고리 보강
- 실패 원인 분류 체계화

이 구조를 통해 AI 출력물을 그대로 믿지 않고, 자동 검증 후 사람이 승인하도록 만들었다.

---

## 9. LLM Provider 확장

초기에는 Anthropic Claude 중심이었지만, 현재는 여러 모델을 선택할 수 있는 구조로 확장했다.

| 모델 prefix | Provider |
|---|---|
| `claude-*` | Anthropic |
| `gpt-*` | OpenAI |
| `gemini-*` | Google Gemini |
| `gemma-*` | Google 계열 |

이를 통해 비용, 품질, 속도를 비교할 수 있는 기반이 생겼다.

---

## 10. 현재 진행 중인 작업

현재는 Gemini 모델로 실제 게시판 매뉴얼 기반 실행을 진행 중이다.

| 단계 | 상태 |
|---|---|
| Stage 1 | 완료 |
| Stage 2 | 26개 기능 중 22개 완료 |
| Stage 3 | Stage 2 완료 후 실행 예정 |

중단 사유는 코드 문제가 아니라 Gemini 무료 quota 제한이다.

quota가 리셋되면 남은 4개 기능을 이어서 실행할 수 있다.

---

## 11. 남은 과제

다음 단계에서 확인해야 할 것은 다음과 같다.

- Gemini 실행 재개 후 Stage 3 검증 완료
- 그누보드5 대상 end-to-end 실행
- Docker와 PostgreSQL 환경 준비
- 실제 시험 업무 기준에 맞는 결과물 검토
- Provider별 품질과 비용 비교

즉, 다음 목표는 **Mock 검증에서 실제 OSS 검증으로 확장**하는 것이다.

---

## 12. 중간 결론

AWT는 현재 다음 상태에 있다.

- PoC로 기본 가능성 검증 완료
- 데스크탑 앱 구조와 실행 파이프라인 구현 완료
- AI 출력 품질을 검증하는 장치 보강 완료
- 실제 OSS 대상 실전 검증 진입 중

따라서 이번 중간 점검의 결론은 다음과 같다.

**AWT는 개념 검증을 넘어, 실제 시험 대상에 적용 가능한지 확인하는 단계로 진입했다.**

---

## 발표 시 강조할 한 문장

> AWT의 핵심은 AI가 테스트케이스를 대신 쓰는 것이 아니라, AI가 만든 결과를 검증 규칙과 사람 검토를 거쳐 실제 실행 가능한 시험 자산으로 바꾸는 것이다.
