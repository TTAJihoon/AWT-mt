## §1. 용어 설명 (이거 먼저 읽기 — 1분)

### "기법" (design_technique)이란?

같은 기능(예: 글쓰기)에 대해 **어떤 시험 의도로 TC를 만들었나**를 표시. AWT가 한 기능에 happy_path 1개만 만들지 않고 7가지 의도를 골고루 시도하도록 강제 (기존 파일럿의 CRUD 편향 보완 목적).

| 기법 (한국어) | 의도 | 예시 |
|---|---|---|
| 정상 흐름 (happy_path) | 정상 동작 확인 | 정상 입력 → 정상 결과 |
| 등가분할 (equivalence) | 입력 도메인별 동등 처리 | 영문 / 숫자 / 혼합 |
| 경계값 (boundary) | 경계 입력 처리 | 1자, 50자, 51자, 빈문자 |
| 기본 부정 (negative_basic) | 잘못된 입력 거절 | 빈 입력, 권한 없음 |
| 심층 부정 (negative_deep) | 에러 메시지·권한 우회 검증 | 메시지 4축(위치·한국어·이해성·보안), URL 직접 권한 우회 |
| 상태 전이 (state_transition) | 특정 상태에서만 동작 | 로그인 시에만 글쓰기 버튼 노출 |
| 기능 결합 (cross_feature) | 다른 기능과 연계 | 글 작성 → 목록 → 상세 일관성 |

→ TC를 볼 때 "기법" 표시는 *"이 TC가 어떤 의도로 만들어졌나"*를 알려주는 라벨.

### "근거(source_quote)"란?

이 TC의 기대 출력 값이 **매뉴얼의 어느 문장에 근거**했나의 인용.

- **매뉴얼 직접 인용:** 신뢰도 ↑. *"매뉴얼에 이렇게 적혀 있으니 expected가 이렇다"*가 명확.
- **`INFERRED: ...`:** 매뉴얼에 직접 안 적혀있어서 AWT가 *추론*함. 너가 *도메인 판단*으로 확인 필요.

→ 너가 *"왜 이 TC가 이런 expected를 가지나"*를 5초에 검증 가능. INFERRED 항목은 *AWT의 추론이 무리한지 너가 판단*하는 자리.

### "신뢰도(gen_confidence)"란?

AWT가 *이 TC를 자기가 잘 만들었다고 생각하는 정도* (0.0~1.0). source_quote 명료성과 기능 명세 강도 등으로 산정.

- **0.85+** → 매뉴얼 직접 근거, 거의 확실. *빠르게 훑고 [A] 가능*
- **0.50~0.85** → 합리적 추론. *천천히 확인*
- **< 0.50** → 추정 의존. *너의 도메인 판단 필요*

## §2. 결정 방법

각 TC 옆 `결정:` 자리에 4가지 중 하나:

- **`[A] approved`** — 그대로 OK, 자동 실행으로 보냄
- **`[E] edited`** — 일부 수정 후 (사전입력 또는 기대값 다듬기 필요)
- **`[R] rejected`** — 폐기 (사유 한 줄)
- **`[P] pending`** — 결정 보류 (이유 노트)

---

## §3. TC 일람 (10개 기능 × 41 TC)

각 TC = 1 블록. 색상은 우선순위 표시:
- 🟢 = 신뢰도 ≥ 0.85, 매뉴얼 직접 인용 → 빠르게 [A]
- 🟡 = 신뢰도 0.50~0.85, 또는 INFERRED + 결함 샘플 매칭 → 천천히 검토
- 🔴 = 신뢰도 < 0.50, 추정 의존 → 너의 판단이 가장 가치 있음
