## 2. 개선안 (3개 결정)

### D54 — Leaf 그룹핑 최적화 (P3 해소 + 호출 90%↓) · 핵심

**아이디어**: TC_DESIGN을 **leaf 1개**가 아니라 **관련 leaf 묶음(그룹)** 단위로 호출한다. 그러면 LLM이 같은 화면의 기능 관계를 보고 ① 일관된 happy_path, ② cross-feature TC, ③ 중복 제거를 해내며, 호출 수도 급감한다.

**그룹핑 신호 — 실측 비교** (정제 후 leaf 2,518개 기준):

| 신호 | 그룹 수 | 호출(cap12) | 감소 | 관계 응집 |
|---|---|---|---|---|
| per-leaf (현재) | 2,518 | 2,518 | — | ❌ 고립 |
| (대분류, 중분류) | 1,205 | 1,216 | 52% | △ mid도 파편화(690 싱글톤) |
| **source_url (페이지)** | **89** | **247** | **90%** | ✅ **같은 화면 = 관련 기능** |

→ **source_url(페이지) 그룹핑 채택**. 근거: 모든 leaf가 source_url 보유(빈값 0%), 페이지당 median 28개(3~49), 같은 페이지 기능은 본질적으로 관련(같은 폼·플로우). D51 이후 헤더 로그인 등 전역은 `__global__` 그룹으로 분리되어 페이지 그룹이 더 깨끗.

**설계**:
- 그룹 키 = `source_url`(병합 leaf는 primary source). `__global__`은 자체 그룹.
- 과대 페이지는 **cap(기본 12) 단위로 서브배치** → 토큰 예산 보호.
- `TC_DESIGN` 프롬프트를 **복수 leaf 입력**으로 확장:
  - 입력: 그룹의 leaf 목록(분류+implicit_spec) + 페이지 맥락.
  - 출력 규칙: **각 leaf마다 최소 happy_path 1개 보장**(커버리지 손실 0) + 그룹 내 **cross-feature TC**(예: 아이디+비번 조합) + 그룹 전체 TC 수 상한.
- TC ID·분류는 각 TC의 소속 leaf에서 부여(기존 stage2 로직 유지).

**호출 수**: 2,518 leaf → **~247 그룹 호출**(consolidation 후엔 더 적음). ⇒ P3·호출수 동시 해소.

#### D54-A: 페이지 그룹 설계 (위)
화면 내(같은 페이지) 기능 관계 → 그룹 호출.

#### D54-B: 교차 페이지 플로우 패스 (cross_feature) — **1차 포함 확정(Q2)**
페이지 그룹은 *화면 내* 관계만 본다. 페이지를 넘나드는 사용자 여정(로그인→글쓰기→목록 노출 등)은 별도 패스로 설계한다.
- **입력 = 사이트맵 요약**: 통제 어휘(D52) 대분류별로 묶은 "페이지 → 핵심 기능명" 목록(이름만, 분량 작게). 전체 leaf 본문이 아니라 요약이라 토큰 작음.
- **요청**: "대표적 사용자 여정을 식별하고, 각 여정을 검증하는 end-to-end 시나리오 TC를 설계하라. 단계(steps)에 어느 페이지·기능을 거치는지 명시."
- **출력**: 다단계 시나리오 TC — `design_technique = cross_feature`, precondition·steps가 여러 기능을 연결. 주된 기능에 귀속(D19 기존 규칙).
- **상한**: 여정 수 cap(예: 도메인 묶음당 N개) → 폭증 방지.
- **호출 수**: 도메인 묶음 단위 **소수(~5–10회)** — 전체 시간에 거의 영향 없음.
- **신규 contract** `TC_FLOW`(또는 TC_DESIGN 변형)로 분리해 일반 TC 설계와 프롬프트 충돌 방지.

**meta.json**: `grouping = {signal:"source_url", groups, cap, calls}`, `cross_flow = {journeys, calls}`.

### D55 — LLM 호출 병렬화 (P1 해소) · 승수 효과

**아이디어**: 서로 독립인 호출을 **동시 실행**(bounded concurrency). 그룹(D54)·DOM_SPEC 청크·V10 gap·Stage 6 실패분석은 모두 독립.

**설계**:
- `ThreadPoolExecutor(max_workers=CONCURRENCY)` — 기본 `CONCURRENCY=6`(RunConfig 노출).
- 적용: Stage 2 그룹, Stage 0 페이지/청크, V10 보완 gap, Stage 6 실패.
- **결정성**: 결과를 입력 순서로 재정렬(TC ID 안정). 진행 로그는 완료 순.

**스레드 안전성 (필수 설계 포인트)**:
- `llm_client._last_call_time`(RPM) → `threading.Lock` 보호.
- LLM 로그 파일 기록 → 호출별 고유 파일명(이미 timestamp 기반)이라 충돌 없음, 단 카운터(`llm_call_count`)는 Lock 또는 원자적 누적.
- 캐시 dict 접근 → Lock.
- `progress_cb`(GUI) → 스레드→메인 마샬링 필요(시그널/큐). UI 단계에서 검증.

**Rate limit 대응**: bounded concurrency + 기존 retry/backoff(`llm_client.py` 5회·지수)로 429 흡수. Gemini는 `_MIN_INTERVAL`을 동시성 인지형으로(토큰버킷) 보정. 상용(Claude/OpenAI)은 동시 요청 허용.

**효과**: 247 그룹 호출을 동시성 6으로 → 직렬 등가 ~41. V10·Stage6도 동일 배율.

### D56 — V10 보완 재설계 (P2 해소 + TC 폭증 억제) · 단일 최대 레버

**현재**: gap(leaf)마다 TC_DESIGN 1회 순차 = **777회**, TC 484→4,888(10배). 시간·검수 부담 모두 과대.

**재설계 (단계적)**:
1. **선제 충족** — D54 그룹 설계 프롬프트에 negative 카테고리(D49) 요구를 **처음부터 포함** → 1차 통과로 커버리지 충족 → **gap 자체를 대폭 감소**.
2. **gap 배치** — 남은 gap을 **페이지 그룹 단위로 묶어** 호출(leaf마다 따로가 아니라).
3. **병렬화**(D55) — 남은 gap 호출 동시 실행.
4. **증식 상한** — leaf당 V10 추가 TC 수 상한(검수 가능 규모로). 현재 평균 ~5.7개/gap은 과다.

**효과**: 777 순차 → (선제 충족으로 gap↓) × (배치) × (병렬) ⇒ 한 자릿수~수십 회 등가. TC 총량도 합리화.

**주의**: D49(negative ≥60% 충족, 카테고리당 ≥1 TC)의 **커버리지 보장은 유지** — 줄이는 건 "충족 방식의 효율"이지 기준이 아니다.

## 4. 회귀·안전 원칙

- **커버리지 손실 0**: D54 그룹 설계는 leaf당 happy_path 보장. D56은 D49 기준 유지.
- **옵트아웃**: `CONCURRENCY=1`이면 순차(기존 동작). 그룹핑·V10 재설계도 플래그로 끄기 가능.
- **결정성**: 병렬 결과는 입력 순 재정렬 → TC ID·출력 재현성 유지(D38).
- **추적성**: grouping/concurrency/V10 재설계 결과를 meta.json에 기록.
- **단계적 검증**: D54 → D56 → D55 순으로 독립 커밋 + 재실행 시간/커버리지 비교.
