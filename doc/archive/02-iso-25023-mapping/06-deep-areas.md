# Layer 3 심층 시험 운영 가이드

**대상 독자:** 시험원
**위치:** L3 영역의 실제 수행 단계 — AWT는 직접 시험하지 않으므로 본 문서는 *시험원이 자기 작업을 빠르게 시작하기 위한 양식·체크리스트 모음*.

> 본 문서는 *프레임워크 + 빈 양식* 중심. 시험소 실무에 맞춰 시험원이 점진 보완.

---

## 1. 사용성 평가 (Heuristic Evaluation)

### 1.1. Nielsen 10원칙 체크리스트

각 원칙별로 *시험 대상 화면에서 위반 사례 0~N개* 기록:

| # | 원칙 | 위반 사례 | 심각도 (0~4) | 개선 권고 |
|---|---|---|---|---|
| 1 | 시스템 상태 가시성 | | | |
| 2 | 시스템·실세계 일치 | | | |
| 3 | 사용자 통제·자유 | | | |
| 4 | 일관성·표준 | | | |
| 5 | 오류 예방 | | | |
| 6 | 회상보다 인식 | | | |
| 7 | 유연성·효율성 | | | |
| 8 | 미니멀리즘 | | | |
| 9 | 오류 인식·진단·복구 | | | |
| 10 | 도움말·문서 | | | |

심각도: 0=문제 아님, 1=cosmetic, 2=minor, 3=major, 4=catastrophic.

### 1.2. SUS (System Usability Scale) 양식

실사용자 panel(5~10명 권장)에 배포:

10문항 (한국어 표준 번역 사용):
1. 이 시스템을 자주 사용하고 싶다
2. 이 시스템은 불필요하게 복잡하다
3. 이 시스템은 사용하기 쉽다
4. 도움 없이는 사용할 수 없을 것 같다
5. 다양한 기능들이 잘 통합되어 있다
6. 이 시스템엔 일관성이 없는 부분이 너무 많다
7. 대부분 사용자는 빨리 익힐 수 있을 것이다
8. 사용하기 어색하다
9. 이 시스템을 사용하는 데 자신 있다
10. 시작 전에 많은 것을 배워야 한다

5점 척도. SUS 점수 환산식 (표준):
```
점수 = ((Q1-1) + (5-Q2) + (Q3-1) + (5-Q4) + (Q5-1) + (5-Q6) + (Q7-1) + (5-Q8) + (Q9-1) + (5-Q10)) * 2.5
```
- 68점 이상: 평균 이상
- 80점 이상: 우수

### 1.3. AWT가 보조 제공 가능

- 평가 대상 화면 스크린샷 모음
- 매뉴얼에서 추출한 *기능 목록* (Nielsen 4번 일관성 비교용)
- 작업 흐름 다이어그램

## 2. 접근성 심층 시험

### 2.1. 스크린리더 실사용

체크 항목:
- [ ] 페이지 로드 후 *주요 콘텐츠로 빠른 이동* 가능 (skip link)
- [ ] 모든 인터랙티브 요소가 *명명*되어 있음 (label, alt, aria-label)
- [ ] 폼 입력 시 *현재 위치·오류*가 음성으로 전달됨
- [ ] 동적 콘텐츠 변경 시 *알림*됨 (aria-live)
- [ ] 표·목록의 구조가 *의미 있게* 읽힘

권장 도구: NVDA (Windows), VoiceOver (macOS).

### 2.2. 키보드 only 워크플로

체크 항목:
- [ ] 마우스 없이 *주요 작업 5종* 완주 가능
- [ ] Tab 순서가 *시각적 순서와 일치*
- [ ] Focus indicator가 *모든 요소에 명확히 보임*
- [ ] Modal/Dialog가 *focus trap*을 정확히 처리
- [ ] Esc로 modal 닫기 등 *표준 단축키* 작동

### 2.3. 색대비·시각

- WCAG 2.1 AA 기준 대비비 (4.5:1, 큰 글자 3:1)
- 색약·색맹 시뮬레이션 (Chrome DevTools → Rendering)
- 200% 확대 시 *내용 잘림 없음*

### 2.4. AWT 보조

- axe-core 자동 검사 결과 (L2)
- 페이지별 *대비비 측정 후보 영역* 자동 식별
- 키보드 only 워크플로의 *주요 작업 후보 5종* 제안

## 3. 부하·용량 시험

### 3.1. 시나리오 양식

| 시나리오 | 동시 사용자 | 지속 시간 | 측정 지표 | 임계 |
|---|---|---|---|---|
| 일반 부하 | (예) 100 | 10분 | 응답시간 p95, 오류율 | p95 < 2초, 오류 < 1% |
| 피크 부하 | 500 | 5분 | 동상 | p95 < 5초, 오류 < 5% |
| 스파이크 | 0→1000 in 30s | 2분 | 복구 시간 | 1분 내 복구 |
| 장시간 | 50 | 24시간 | 메모리 누수, 오류 누적 | 메모리 평탄, 오류 누적 없음 |

### 3.2. 도구

- **k6** (JavaScript 기반, CLI)
- **Locust** (Python, 분산 가능)
- **JMeter** (GUI, 보수적)

선택 기준: 시험원 친숙도·환경 제약 우선.

### 3.3. AWT 보조

- 부하 대상 기능 식별 (검색·결제·업로드 등)
- 시나리오 prompt 후보
- 제품 설명서에 명시된 *동접 임계*가 있으면 추출 제공

## 4. 보안 심층 시험

### 4.1. 영역별 체크

| 영역 | 점검 항목 (예시) | 도구 |
|---|---|---|
| 인증 | 비밀번호 정책, 세션 만료, 2FA, 무차별 대입 | Burp Suite |
| 인가 | 권한 매트릭스, IDOR, 수직·수평 권한 상승 | 수동 |
| 입력 검증 | XSS·SQLi·CSRF·CRLF | OWASP ZAP |
| 데이터 보호 | TLS 강도, 쿠키 secure/HttpOnly, 민감 정보 노출 | testssl.sh |
| 로깅 | 보안 이벤트 기록, PII 마스킹 | 수동 검토 |

### 4.2. 별도 표준

- OWASP Top 10 (웹 표준 영역)
- OWASP ASVS (수준별 요구)
- ISO/IEC 27001 (정보보안 관리)

본 문서의 보안 부분은 *체크리스트 시작점*이고, 실제 보안 시험은 위 표준 참조.

### 4.3. AWT 보조

- 인증·인가 흐름 다이어그램 (매뉴얼 + 시험 결과 기반)
- 권한 매트릭스 자동 추출
- L2 보안 기초 점검 결과 (출발점)

## 5. 외부 통합 실거래 시험

### 5.1. 식별

AWT가 제공 가능:
- 결제 PG·SMS·메일·외부 API 호출 지점 자동 식별
- 각 통합 지점의 mock vs 실거래 옵션

### 5.2. 시험 방침

- *원칙: 실거래 시험은 sandbox 환경에서.*
- 실서버 시험은 *위험성 별도 결재* 후 수행
- 실거래 시험 결과는 *별도 결함 리포트*로 분리

### 5.3. 회귀 시험 자산화

성공한 실거래 시험 절차는 *문서화해 재사용*. AWT가 이를 *RAG에 익명화 축적* 가능 (D16).

## 6. L3 결과의 통합 기록 양식

각 영역별 시험 종료 시 다음 양식으로 기록:

```
영역: [사용성·접근성·부하·보안·외부통합 중 선택]
시험원: [reviewer_id]
시험 일자: YYYY-MM-DD
시험 소요: ___ 시간

[수행 항목]
- 항목 1: 결과 요약
- 항목 2: ...

[발견 결함]
- 결함 1: 심각도, 재현 절차, 권고
- 결함 2: ...

[종합 평가]
[보완 필요 사항]
```

본 양식은 AWT가 *빈 채로 제공* → 시험원이 채움 → 최종 보고서 §4에 포함.

## 7. AWT의 *L3 보조 자료 출력 시점*

L3 보조 자료는 *Phase 1 자동 시험 종료 직후* 별도 폴더에 출력:

```
output/
├─ Layer1_AutoResults.xlsx
├─ Layer2_GateResults.xlsx
└─ Layer3_Aids/
   ├─ heuristic_screenshots/
   ├─ accessibility_axe_report.html
   ├─ load_scenario_candidates.md
   ├─ security_flow_diagram.png
   ├─ integration_points.md
   └─ tester_form_templates.xlsx
```

시험원은 *Layer3_Aids/* 폴더 내용을 출발점으로 본인 작업 시작.
