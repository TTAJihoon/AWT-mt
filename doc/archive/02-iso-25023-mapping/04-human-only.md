# Layer 3 — 시험원 전담 상세

**적용 범위:** `01-characteristics-matrix.md`에서 L3로 분류된 메트릭.
**책임 주체:** **시험원 100%.** AWT는 직접 시험을 수행하지 않으며 *유관 정보·자료만 정리·제공*.

> **L3의 본질:** AWT가 *대체할 수 없는* 영역. 사용자 인지·도메인 지식·표준 외 작업이 필수. AWT의 가치는 *시험원이 이 영역에 집중할 수 있도록 L1·L2 작업 시간을 줄여주는 것*.

---

## 1. L3 영역 목록 (요약)

| 영역 | 25010 부특성 | AWT의 보조 가능 자료 |
|---|---|---|
| 사용성 인지·학습성 | Appropriateness recognizability, Learnability | 사용자 문서 추출, 작업 흐름 다이어그램 |
| UI 미적 | User interface aesthetics | 스크린샷 묶음, 디자인 토큰 추출 |
| 부하·용량 | Capacity | 해당 기능 식별, 부하 시나리오 후보 제안 |
| 보안 심층 | Confidentiality·Integrity 심층, Non-repudiation, Accountability | 인증 흐름 다이어그램, 권한 매트릭스 추출 |
| 유지보수성 (블랙박스 N/A) | Modularity, Reusability, Analyzability, Modifiability | 보통 N/A 또는 별도 소스 시험 |
| 이식성 (Installability·Replaceability) | Installability, Replaceability | 환경 요구 추출, 데이터 마이그레이션 식별 |
| 외부 통합 실거래 | (해당 없음 — 분류 외) | 통합 지점 식별, mock vs 실거래 옵션 |

## 2. AWT가 L3에 *제공할 수 있는 보조 자료*

L3는 시험원 전담이지만 AWT가 *시험 준비 시간*을 줄여줌:

### 2.1. 사용성 평가 자료

- **작업 흐름 다이어그램 자동 추출** — 매뉴얼·DOM 스캔으로 주요 기능의 user journey를 시각화
- **screen shots 묶음** — 핵심 화면 N개를 스크린샷으로 저장 → 시험원의 heuristic evaluation 자료
- **Nielsen 10원칙 체크리스트 템플릿** — 시험원이 직접 채워가는 양식 제공
- **SUS·USE 설문 양식** — 시험원이 실사용자 panel에 배포할 때 사용

### 2.2. 미적 적합성 자료

- 디자인 토큰 추출 (색상 팔레트, 폰트, 간격)
- 페이지별 layout breakpoint 스크린샷 (PC/태블릿/모바일)
- AWT의 *판정 시도 없음*. 자료 정리만.

### 2.3. 부하·용량 시나리오 후보

- 부하가 예상되는 기능 자동 식별 (예: 검색, 결제, 대용량 업로드)
- 부하 시나리오 prompt 후보 (시험원이 k6/Locust 등으로 직접 실행)
- 추정 동시 사용자 임계 후보 (제품 설명서 명시값이 있으면 참조)

### 2.4. 보안 심층 보조

- **인증·인가 흐름 다이어그램** — 로그인 → 권한 → 자원 접근 흐름
- **권한 매트릭스 추출** — 어떤 role이 어떤 기능에 접근 가능한지 매뉴얼·시험 결과 기반 자동 추출
- **AWT 기초 점검 결과** (L2의 Security 기초 결과) — 시험원이 심층 시험 시 *참고 출발점*
- *AWT의 보안 판정은 신뢰하지 말 것* — 보안 전문가의 도구·방법론 필수

### 2.5. 설치·교체 시험 자료

- 시스템 요구사항 추출 (제품 설명서에서)
- 데이터 마이그레이션 시나리오 식별
- AWT의 *시험 수행 없음*. 시험원이 환경 셋업 후 직접 수행.

## 3. L3 결과의 *시험 보고서 통합*

L3 시험 결과도 최종 산출 보고서에 들어가야 함. AWT는 *통합 양식*을 제공:

```
시험 보고서 구조 (제안):
├─ §1. 시험 개요 (자동)
├─ §2. Layer 1 자동 시험 결과 (AWT 자동 채움)
├─ §3. Layer 2 보조 + 검토 결과 (AWT + 시험원)
├─ §4. Layer 3 전담 시험 결과 (시험원 직접 작성)
│   ├─ §4.1 사용성 평가 (heuristic + SUS)
│   ├─ §4.2 부하·용량
│   ├─ §4.3 보안 심층
│   └─ §4.4 (기타 해당 시)
└─ §5. 통합 결함 리포트 (모든 Layer)
```

§4의 각 절은 *시험원이 직접 채우는 빈 양식*을 AWT가 제공.

## 4. L3와 L2의 경계 사례 (애매한 영역의 가이드)

| 사례 | L2? L3? | 가이드 |
|---|---|---|
| WCAG 자동 점검 결과 + 스크린리더 실사용 | 자동 점검은 L2, 스크린리더는 L3 | 두 결과를 분리 보고. 합산 평가 시 시험원 판정 |
| 에러 메시지의 위치는 적절한가 | L2 (negative_deep 4축) | AWT 의견 제시 → 시험원 승인 |
| 에러 메시지의 *한국어 자연스러움* | 경계 — L2가 시도하되 confidence ↓ | 시험원이 항상 검토. 또는 *처음부터 L3*로 분류 가능 |
| 부하 1000 사용자 동접 시 응답시간 | L3 (Capacity) | AWT는 시나리오·후보만 |
| 동일 기능을 100회 연속 실행 시 안정성 | L1 또는 L2 (Maturity·Reliability) | 자동 가능 |

→ 애매한 영역은 *위험 큰 쪽 (L3 또는 더 높은 인간 개입)*으로 분류하는 게 보수적 안전.

## 5. L3 시간 분배 권장 (시험원 처리량 관점)

AWT가 L1·L2의 시간을 단축한 *남는 시간*을 L3에 투자하는 게 본 프로젝트의 시험소 가치 명제:

```
시험원 시간 배분 (제품 1개당, 가설):
[Before AWT]                       [After AWT (Phase 1 안정화 후)]
  L1: 60% (수동 실행·기록)       →    L1: 5%  (Gate 확인)
  L2: 30% (보조 + 판정)          →    L2: 35% (집중 검토)
  L3: 10% (시간 부족으로 축소)   →    L3: 60% (충분한 시간 확보)
```

L3 시간이 늘면:
- 더 깊은 사용성 평가
- 더 정밀한 보안 시험
- 더 다양한 도메인 시나리오 탐색
- *전문가로서의 가치 발휘*

→ 사용자 표현 "*나머지 부분은 사람이 할 수 있다*"의 실현.

---

## 6. AWT가 L3에 *손대지 말아야 할 선*

다음은 AWT가 *판정·결론을 내지 않아야* 함 (자료 정리에 머물 것):

- *이 UI가 아름다운가* 같은 미적 판정
- *사용자가 이 기능을 학습하는데 얼마나 걸리는가* 같은 인지 판정
- *이 보안 점검이 충분한가* 같은 보안 충분성 판정
- *이 제품이 인증 통과 자격이 있는가* 같은 종합 판정 (D6: AI는 *시험원 결정 지원*이지 *결정 자체*가 아님)

L3 영역에서 AWT의 *판정 시도*는 해당 산출을 자동으로 INFERRED + 시험원 우선 검토로 강등.
