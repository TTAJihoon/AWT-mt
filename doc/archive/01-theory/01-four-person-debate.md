# 4인 토론 — AI 에이전트 기반 웹 자동 시험의 신뢰성·커버리지·잔여 인간 검증 범위

**일시:** 2026-05-18
**맥락:** AWT 프로젝트 착수 전 이론적 방향성 검토
**전제:** ISO/IEC 25023(품질측정 메트릭) 기준의 SW 시험인증. 입력은 매뉴얼 + 기능리스트 + URL 기반 DOM/이미지 자동 스캔. AI 에이전트가 TC 설계 → 자동 실행 → 결과 판정까지 수행.

## Round 1 — "AI가 만든 TC가 사람이 만든 TC와 비교해 커버리지를 얼마나 확보하나?"

**B (낙관):** 입력이 매뉴얼·기능리스트·DOM·이미지 4종이면, **기능적 적합성(Functional Suitability)** 영역에서 인간 테스터 평균 커버리지의 80~95%는 도달한다고 본다. 특히 *Functional completeness*(명세된 기능 중 시험된 기능 비율)는 기능리스트가 그대로 입력되니 거의 100%에 수렴한다. 사람은 빠뜨리고, AI는 빠뜨리지 않는다.

**A (회의):** 그 수치가 함정이다. 25023의 *Functional completeness*는 "**명세된 기능을 시험했는가**"가 아니라 "**시험 시나리오가 그 기능의 의도된 거동을 측정했는가**"다. AI가 "로그인 버튼 클릭 → 대시보드 진입"이라는 1개 TC를 만들면 함수 단위로는 커버됐지만, **잘못된 비밀번호·계정 잠금·세션 만료·동시 로그인**까지 안 만들면 *Functional correctness* 측정엔 빈약하다. **커버리지 수치만 보면 안 되고 등가분할/경계값/상태전이 분할 충실도를 봐야 한다.**

**D (방법론):** A의 말이 정확하다. **3계층 커버리지 모델**로 분리해서 봐야 한다:
1. **요구사항 커버리지** — 기능리스트 vs TC 매핑 (AI가 강함, 95%+)
2. **입력 도메인 커버리지** — 등가분할·경계값·invalid input (AI가 중간, 60~75%, *매뉴얼에 명시되지 않은 경계는 추론*)
3. **상태/시나리오 커버리지** — 다단계 워크플로, 상태 전이, 비즈니스 룰 (AI가 약함, 40~60%)

**C (실용):** 3번이 핵심이다. 예: 보험견적 사이트에서 "갱신 회원 + 30일 내 사고이력 + 특약변경"이 동시 만족할 때만 발생하는 가격 산정 버그. 이건 매뉴얼에도 DOM에도 안 나온다. **도메인 지식 없이는 못 만든다.** AI는 "기능당 1~3개 정상/예외 TC"는 잘 뽑지만, **기능 간 결합으로 생기는 시나리오**는 거의 못 잡는다.

**B (반론):** 그건 prompt에 "기능 간 cross-feature 시나리오를 5개 이상 만들라"고 명시하거나, 기능리스트에서 N×N 조합을 sampling하면 보완된다. 또 pairwise testing 알고리즘(IPOG 등)을 결합하면 D가 말한 3번 커버리지도 65~80%까지 끌어올릴 수 있다.

**A (재반론):** 끌어올릴 수 있어도 **그게 의도된 비즈니스 룰을 시험하는지**가 증명되지 않는다. pairwise로 조합해도 "그 조합이 실제 비즈니스 의미가 있는가"는 매뉴얼만 보고는 모른다.

> **Round 1 합의:** 요구사항·기능 커버리지는 AI가 인간 동등 이상. 입력 도메인 커버리지는 보강 가능. 시나리오·비즈니스룰 커버리지는 **구조적 한계** — 도메인 지식 주입 없이는 인간 대체 불가.

## Round 2 — "자동화하기 어려운 영역은 어디인가?"

**C (실용):** 현장 경험으로 7가지 영역:
1. **시각적 적합성** — 레이아웃 깨짐, 색대비, 가독성 (이미지 스캔으로 일부 가능하나 *의도된 디자인 vs 버그* 구분 불가)
2. **접근성(WCAG)** — axe-core 같은 자동도구는 30~40%만 검출 (W3C 통계)
3. **사용성(Usability)** — 25023의 *Learnability, Operability, User error protection* — 정의상 인간 인지 필요
4. **콘텐츠 정합성** — 약관·고지·법적 문구 정확성
5. **동시성/경합** — 멀티유저, 동시 결제, 재고 차감
6. **장기 상태** — 세션 24시간, 토큰 만료, 캐시 stale
7. **외부 통합** — 결제 PG, SMS, 메일 (mock 없이 실거래 테스트)

**B (낙관):** 1·2는 컴퓨터 비전 + 접근성 트리 분석으로 점진 개선 중. 5·6은 부하 도구(k6, Locust) 연계로 자동화 가능. 7은 환경 구성 문제지 자동화 한계는 아니다.

**A (회의):** 3·4가 인증 관점에서 **가장 결정적**이다. 25023의 *Appropriateness recognizability*, *User interface aesthetics*, *Accessibility*는 **인간 평가자 panel** 또는 **표준화된 설문(SUS, USE)** 없이는 정량화 자체가 안 된다. AI가 시뮬레이션해도 "AI가 사용자 역할극 한 결과"라 25023 측정의 *operational definition*을 만족하지 못한다.

**D (방법론):** 추가로 **oracle problem**. AI가 TC를 실행해서 결과를 봤는데 "그 결과가 PASS인가 FAIL인가"를 판단할 *oracle*이 약하다. 매뉴얼에 "정상 동작" 같이 모호하게 적힌 부분은 AI가 자체 oracle을 만들어버린다 — 이게 **silent false negative**의 주범. 메타모픽 테스팅(입력 변환에 따른 출력 관계 검사)이 oracle 부족 문제의 학술적 해법이지만 적용 사례는 아직 제한적.

**C:** 한 가지 더 — **negative path의 깊이**. AI는 보통 "잘못된 입력 → 에러 메시지 확인" 한 단계로 끝낸다. 실제 테스터는 *"에러 메시지가 적절한 위치에, 적절한 한국어로, 비기술 사용자가 이해 가능하게, 보안정보 노출 없이"* 4중 확인한다. AI는 첫 단계만 본다.

> **Round 2 합의:** 자동화 한계 영역 = ①사용성·미적·인지 ②접근성 깊이 ③비즈니스 콘텐츠 정합성 ④동시성·장기상태 일부 ⑤**Oracle 신뢰성** ⑥negative case의 깊이.

## Round 3 — "AI가 생성한 TC의 설계와 결과를 어디까지 믿을 수 있는가?"

**A (회의·강함):** 인증의 핵심은 **trust artifact의 무결성**이다. 5가지 신뢰 축으로 분해한다:
1. **TC 설계 의도의 추적성** — 각 TC가 어느 요구사항에서 파생됐는지 (matrix)
2. **재현성** — 같은 입력 → 같은 TC (LLM 비결정성 문제)
3. **결과 판정 근거** — PASS/FAIL이 어떤 oracle로 정해졌는지
4. **누락 검출 능력** — 못 만든 TC가 무엇인지 알 수 있나
5. **편향** — 학습 데이터 기반의 일반적 TC만 만들고 *제품 특이성* 못 잡는 편향

LLM은 **2번에서 구조적으로 약하다.** Temperature 0이어도 모델 업데이트되면 깨진다.

**B (낙관·조건부):** 2번은 seed 고정 + prompt hash + 모델 버전 pinning + TC 산출물을 git에 commit하면 *artifact 수준의 재현성*은 확보된다. **"매번 같은 TC 생성 가능"이 아니라 "한 번 생성한 TC를 동결·재실행 가능"**이 인증에서 더 중요하다.

**D (방법론):** B의 재정의가 옳다. **소프트웨어 공학 표준 관행으로 보면 TC는 산출물이지 함수가 아니다.** TC를 한 번 만들면 그건 코드처럼 버전 관리되는 자산이고, 인증은 그 자산에 대한 검증이다. 다만 *"이 자산을 어떻게 도출했는가"*에 대한 **methodology audit**이 필요하다 — 이게 ISO/IEC/IEEE 29119에서 요구하는 부분.

**C (실용):** 결과 신뢰성에서 내가 본 흔한 거짓양성/거짓음성:
- **False PASS**: AI가 클릭했는데 페이지가 안 바뀜 → "이미 그 페이지에 있어서 OK"로 판정 (실제론 클릭 미동작)
- **False FAIL**: 페이지 로딩 1초 지연 → "expected text 없음" → FAIL (실제 기능 정상)
- **Hallucinated assertion**: 매뉴얼에 없는 거동을 AI가 "마땅히 그래야 한다"고 추론해서 assert

특히 3번이 무섭다 — **인증 통과/불합 판정이 AI 환각 위에 서있게 됨**.

**A:** C의 3번이 결정타다. 인증에서는 *"실패 판정의 근거가 명세에 있는가"*가 ISO 17025 시험소 인정 차원에서도 필수. **모든 assertion은 매뉴얼/기능리스트의 특정 항목으로 trace 가능해야 한다.** AI가 만든 TC는 이 traceability를 강제하지 않으면 폐기해야 한다.

**B:** 그건 prompt 구조로 강제 가능 — 모든 TC는 `{requirement_id, assertion, source_quote}` 3튜플을 반드시 포함하도록. source_quote가 매뉴얼에 grep으로 찾아지지 않으면 reject.

> **Round 3 합의:** AI 산출 TC는 **(a) 동결된 산출물로 취급**, **(b) requirement-to-assertion traceability 강제**, **(c) source quote 검증 가능**, **(d) oracle 근거 명시** 4조건 충족 시에만 인증 증거로 활용 가능. 미충족 시는 *exploratory hint*로만 사용.

## Round 4 — "TC 생성 + 자동 실행 후, 실제 테스터의 잔여 검증 범위는?"

**C (실용):** **3단 필터링**을 제안:
- **Layer 1 — AI 자동 (Bulk):** 기능별 정상/단순예외 TC, 회귀 시험, 링크·DOM 무결성, 기본 접근성 (axe-core 류)
- **Layer 2 — 인간 검토 (Curation):** AI 생성 TC를 *읽고* 누락·환각·근거 부재를 솎아냄. 비즈니스 룰 시나리오 보충. 추정 30~50% 시간 절감의 핵심.
- **Layer 3 — 인간 전담 (Irreplaceable):**
  - 사용성 평가 (heuristic evaluation, Nielsen 10원칙)
  - 접근성 심층 (스크린리더 실사용, 키보드 only 워크플로)
  - 콘텐츠·법적 문구 정합성
  - 도메인 특이 시나리오 (cross-feature)
  - 보안 (인증/인가 우회, IDOR, XSS) — 별도 표준
  - 탐색적 시험 (charter-based)

**A (회의):** Layer 2가 가장 중요하다. **AI 산출물을 그대로 인증 증거로 쓰면 안 된다.** "검토자가 읽고 승인했다"는 서명이 필요. 인증 관점에서는 *AI는 보조 도구, 책임자는 인간*이라는 원칙 유지해야 한다.

**D:** 시험소가 측정해야 할 지표 추가 제안:
- **AI TC acceptance rate** (인간이 그대로 수용한 비율)
- **AI TC augmentation rate** (인간이 수정한 비율)
- **Human-added TC rate** (인간이 추가한 비율)
- **Defect attribution** (실제 발견된 결함이 어느 layer에서 나왔는지)

이 지표를 누적하면 **layer 1의 한계를 정량적으로 학습**하고 다음 prompt/도구 개선에 환류 가능.

**B:** 동의. 추가로 *layer 1의 결과 자체에 confidence score를 붙이는 것*. AI가 TC 실행 후 "이 PASS는 확신도 0.9, 이 PASS는 0.4"식으로 산출하면 layer 2 검토자가 0.4부터 우선 본다 — **risk-based review**.

> **Round 4 합의:** 인간의 잔여 범위 = **(i) AI TC curation/승인, (ii) 사용성·접근성·콘텐츠·도메인·보안의 심층, (iii) 탐색적 시험, (iv) 결함의 layer별 귀속 추적**. AI는 1차 통과를 만들고 인간은 *gate keeper + 깊이 + 책임*.

## Round 5 — "더 좋은 방법을 적용한다면?"

**D (방법론, 연구기반):** 우선순위 높은 5가지:
1. **Model-Based Testing (MBT)** — DOM 스캔 결과로 *상태기계 모델* 자동 생성 → 그 위에서 path coverage 계산. AI가 "TC를 직접 만들기"보다 "**모델을 만들고 모델에서 TC를 도출**"하는 게 학술적으로 안정적.
2. **Metamorphic Testing** — oracle 부재 영역에 적용. 예: 검색 "사과" 결과 ⊇ 검색 "사과 빨강" 결과. AI가 metamorphic relation을 매뉴얼에서 추출하는 연구가 활발.
3. **Mutation-based TC quality 평가** — TC 자체의 강도를 측정. AI 생성 TC가 인간 TC 대비 mutation score 몇 %인지 — 이게 *진짜 커버리지 지표*.
4. **Property-based testing** — 단일 example이 아닌 property를 명시 (예: "어떤 입력이든 N자 이하면 reject"). Hypothesis 류 도구와 결합.
5. **Differential testing** — 같은 기능의 이전 버전과 결과 비교. 회귀 자동 검출.

**B (낙관):** 운영 관점에서 4가지:
1. **Multi-agent 분업** — TC생성 agent / 실행 agent / oracle 검증 agent / 보고서 agent 분리. 단일 agent의 hallucination 상호 검증.
2. **RAG으로 도메인 지식 주입** — 과거 인증한 유사 제품의 TC, 결함 이력을 vector DB에 → 신제품 TC 생성 시 retrieval. **이게 시험소의 가장 큰 자산**.
3. **Self-healing 자동화** — 셀렉터 깨짐 자동 복구 (Playwright + LLM).
4. **Visual regression + LLM judge** — pixel diff가 아닌 "의미적 차이" 판단.

**A (회의·조건부 수용):** D의 1·3, B의 2에 강하게 동의. 특히 **B-2 (RAG으로 결함 이력 주입)이 시험소의 압도적 차별화 요소**. 외부 SaaS AI 도구 대비 *시험소만이 가진 데이터*. 단, 결함 이력은 고객사 비밀 — **익명화·일반화한 패턴 DB**로 구축해야 한다.

**C (실용):** 빠진 게 있다. **결함 재현 자동화** — 사람이 발견한 결함을 AI가 *최소 재현 TC*로 변환해 회귀집합에 자동 편입. 이게 누적되면 시험소가 *살아있는 회귀 자산*을 갖게 된다.

**D:** C의 제안은 *defect-driven test augmentation*. 산업계에서 가장 효과 큰 방법 중 하나로 알려져 있다.

> **Round 5 합의 — 단기/중기 로드맵 후보**
> - **단기 (3~6개월):** Multi-agent 분업, requirement-traceability 강제 prompt, RAG 결함이력 DB 구축 시작, mutation score 측정 파일럿
> - **중기 (6~18개월):** MBT 도입 (DOM→상태기계), metamorphic relation 자동 추출, differential testing 자동화, self-healing selector
> - **장기 (18개월+):** 시험소 누적 결함 DB 기반 *제품군별 specialized agent*, property-based testing 도구화

## 종합 — 도출된 8대 원칙

| # | 원칙 | 근거 라운드 |
|---|---|---|
| P1 | AI TC는 **동결된 산출물**로 관리 (재현성 ≠ 동일 재생성, 동결 재실행) | R3 |
| P2 | 모든 assertion은 **requirement_id + source_quote**로 trace | R3 |
| P3 | 3계층 커버리지 모델로 측정 (요구사항 / 입력도메인 / 시나리오) | R1 |
| P4 | 인간 검증 = **curation + 심층(사용성/접근성/콘텐츠/도메인/보안) + 탐색적 + 책임** | R4 |
| P5 | AI 결과에 **confidence score** 부여 → risk-based human review | R4 |
| P6 | **결함이력 RAG**가 시험소 차별화의 핵심 자산 | R5 |
| P7 | 단일 LLM 대신 **multi-agent 상호 검증** | R5 |
| P8 | mutation score / acceptance rate 등 **TC 품질 자체의 메타 측정** | R4, R5 |

## 토론에서 사용자가 확정한 사항 (2026-05-18)

- Agent 역할 범위: **TC 설계 + 자동 실행 + 결과 판정** (end-to-end)
- 25023 적용: 자동화 가능한 특성부터 모두 적용, 어려운 부분은 분류
- RAG: **익명화 전제로 활용 가능**
- 산출 형태: 설계 문서(doc/) 우선, 구현은 별도 단계
- Skill 전략: 프로젝트 최상위 SKILL.md 1개 + 추가로 분리 배포 가능한 sub-skill 식별
