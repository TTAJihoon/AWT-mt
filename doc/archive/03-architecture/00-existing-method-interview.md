# 기존 시험방법 인터뷰 — As-Is 시스템 이해

**인터뷰 일자:** 2026-05-18
**대상:** 시험소 시험원 (AWT 프로젝트 사용자)
**맥락:** 시험소에서 운영 중인 Claude Code 기반 시험방법 skill의 운영 방식을 파악하기 위한 As-Is 인터뷰. *원래 목적*은 AWT의 강화 대상 파악이었으나, **D25 pivot (2026-05-19) 이후 본 인터뷰는 reference material로 위상 변경.** AWT는 standalone으로 구현되며 기존 skill을 호출하지 않음.

> 사용자가 직접 작성한 skill이 아니며, 직접 실행해본 적도 없음 (결과만 관찰). 시험방법론에 대한 지식은 보유 (D14).
> **본 인터뷰의 가치 (D25 후):** AWT 설계가 *시험소 실무*와 동떨어지지 않도록 하는 *참고 자료*. 9개 사실(F1~F9)은 AWT 입력 형식·산출 양식·강화 우선순위 결정의 근거로 여전히 유효.

---

## A. 입력 자료 형태

- **매뉴얼 형식:** PDF > Word > PPT 순 (빈도)
- **기능리스트:** 시험소 정해진 양식의 Excel 파일, **대분류 / 중분류 / 소분류** 컬럼 구조
- **URL:** 텍스트 입력
- **추가 입력:** **각 ISO/IEC 25023 품질특성별 결함 샘플 문서** 제공 — 이는 사실상 *seed RAG* 역할
- **AI 접근 방식:** 파일이 저장된 폴더에 AI가 직접 접근해 읽어감 (folder-based ingestion)

**시사점:**
- 매뉴얼 파싱은 PDF가 최빈 — PDF 처리 품질이 핵심 (스킬 `anthropic-skills:pdf` 활용 가능)
- Excel 기능리스트가 이미 정형 → TC ID(D9, D17)의 XXX 일련번호 부여가 자연스러움
- **품질특성별 결함 샘플 문서가 이미 운영 중** = AWT의 RAG 설계의 *현행 미니 버전*. 누적하면 곧 본격 RAG

## B. TC 생성 전략 (현행)

- **Prompt 요구사항:** "정상/예외 만들라"는 정도만 요구
- **설계 기법 명시:** *없음* — 등가분할·경계값·상태전이·negative path 등 명시 안 함
- **AI 동작:** 샘플 결함 + 제품 내용을 *자체 추론*해서 TC 생성
- **규모:** 1개 제품당 약 **100~200개 TC**, **주로 CRUD 위주**

**시사점:**
- AI가 자체 추론에 맡겨져 → 자연스럽게 *학습 데이터의 평균 패턴* = CRUD에 수렴
- 사용자 본인 표현: "CRUD 위주로 한 것 같은데 그 외의 기능도 제대로 한건지" → *커버리지 불안*의 근본 원인
- **이게 AWT 강화의 1순위 영역.** Prompt에 TC 설계 기법을 명시적으로 강제하면 즉시 개선 가능

## C. 실행과 결과 판정 (현행)

- **자동 실행:** 클릭·입력까지 자동 수행, 정해진 양식의 결과 보고서까지 작성
- **판정 주체:** **AI가 PASS/FAIL 판정 → 사람이 검토**
- **결과 형식:** 각 TC의 **사전입력 / 기대값 / 실제출력** 3컬럼
- **약점 (사용자 명시):** "왜 실패했는지는 직접 확인해야 함" — **실패 원인 자동 기록 부재**

**시사점:**
- End-to-end 자동 실행은 이미 작동 = 큰 자산 (보존해야 함)
- 그러나 판정의 *근거*가 산출물에 없음 → AI 환각 판정 위험 ↑
- AWT 원칙 P2(source_quote)·P5(confidence score) 도입의 가장 큰 효익이 여기에 있음 — *실패 원인 기록 + 판정 근거 추적*

## D. 산출 보고서

- **파일 형식:** Excel
- **2개 문서 산출:**
  - **결함 리포트** — 결함이 포함된 품질특성 + 발생 상황 설명
  - **TC 문서** — 사전 / 기대 / 결과 형식의 TC표
- **25023 메트릭 수치:** **작성하지 않음** (예: 기능 완전성 %)

**시사점:**
- Excel 출력은 시험소 표준 (스킬 `anthropic-skills:xlsx` 활용 적합)
- 25023 메트릭 %를 *작성하지 않는다*는 사실은 doc/02-iso-25023-mapping 설계에 직접 영향 — **메트릭 계산은 내부 품질 관리 용도이지 외부 산출이 아닐 가능성**. *확인 질문 필요.*

## E. 현재의 한계·아쉬움 (사용자 본인 진술)

1. **신뢰 불확실성** — "만들어진 TC를 믿어야 하는건지" 자체가 답이 없음
2. **커버리지 불확실성** — "CRUD 위주로 한 것 같은데 그 외 기능도 제대로 한건지"
3. **Reviewer 경험 부재** — 사용자 본인이 *아직 reviewer 역할을 해본 적 없음*
4. **희망 사항 (사용자 표현):** "AI가 제대로 된 TC를 만들고 자동화 가능한 부분만 제대로 수행해주면 나머지 부분은 사람이 할 수 있다"

**시사점:**
- 4번이 **AWT의 미션 정의 그 자체**. AWT는 시험원의 이 한 문장을 충족하면 성공.
- 1·2번은 AWT 8대 원칙(특히 P2·P3·P5)으로 직접 해소 가능
- 3번 — Layer 2 워크플로가 *아직 운영 중이 아님*. 즉 AWT는 Layer 2 자체를 *처음 도입*하는 셈. 운영 절차 설계 부담이 큼.

## 응답에서 도출된 사실 (Facts)

| # | 사실 | 영향 |
|---|---|---|
| F1 | 매뉴얼은 PDF/Word/PPT 혼재 | PDF 파싱 품질이 입력 충실성의 bottleneck |
| F2 | 기능리스트는 대/중/소 분류 Excel | TC ID 일련번호 부여 자연스러움 (D17 부합) |
| F3 | 품질특성별 결함 샘플이 이미 제공 중 | RAG의 seed가 이미 운영 중 (Phase 1에서 정리만 하면 즉시 활용 가능) |
| F4 | Prompt에 TC 설계 기법 미명시 → CRUD 편향 | **강화 1순위** — 즉시 효과 큰 개선점 |
| F5 | AI가 PASS/FAIL 판정, 실패 원인은 미기록 | 신뢰 불확실성의 직접 원인 |
| F6 | Excel TC 문서는 사전/기대/결과 3컬럼 | AWT 강화 시 컬럼 추가 (requirement_id, source_quote, confidence, oracle_reason, ...) |
| F7 | 25023 메트릭 %는 산출하지 않음 | 메트릭의 외부 산출 vs 내부 품질관리 용도 — 확인 필요 |
| F8 | Layer 2 reviewer 워크플로는 미존재 | AWT가 처음 도입 — 단순하고 강제력 있는 설계 필요 |
| F9 | 1제품당 100~200개 TC, 주로 CRUD | Reviewer 시간 budget 환산: 100~200 × 30초~2분 = 50~400분 = 약 1~7시간/제품/reviewer |

## 인터뷰에서 파생된 추가 질문 (다음 결정 시점에 답변 필요)

- **Q-INT-1**: 25023 메트릭 %를 *작성하지 않는* 이유 — (a) 시험소가 측정은 하나 보고서에 안 적는다 (b) 측정 자체 안 한다 (c) 별도 내부 문서에만 적는다 중 어느 쪽?
- **Q-INT-2**: 결함 리포트의 "발생 상황"은 자유서술인가, 정해진 필드(예: 재현 절차 / 환경 / 빈도)가 있는가?
- **Q-INT-3**: 현재 AI 산출 TC와 결함 리포트가 *그대로 최종 산출물*이 되나, 아니면 시험원이 한 번 가공·승인 단계가 있나? (현재 reviewer 워크플로 자체가 없다면 더더욱 확인 필요)
- **Q-INT-4**: AI 자동 실행의 도구 (Playwright? 브라우저 자동화? Claude Code의 어떤 MCP?) — Phase 1 신뢰성 확보에 필요한 정보
