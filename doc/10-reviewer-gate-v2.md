## 1. 문제 정의

| # | 문제 | 현 상태(`reviewer_gate.py`) |
|---|---|---|
| #4 | "happy_path 아닌 걸 보면 된다"는 막연함 + **수백 건 다 봐야 할 압박** | TC가 **평면 목록**으로 나열, 우선순위·필터 없음. 진행률이 "pending 484"라 끝이 안 보임 |
| #3 | 검토에 필요한 정보·동선 부족 | 더블클릭 팝업이 단순(스크린샷·전후이동·키보드 없음), 결정 근거 신호 미표시 |

**핵심 통찰(토론 합의)**: Reviewer Gate(D22)의 목적은 *모든 TC 검수*가 아니라 **실행 낭비 전에 거짓·부실 TC를 거르는 것**. 따라서 "기법" 하나보다 TC가 이미 가진 **생성 신뢰도(`gen_confidence`) + 근거 출처(`source_quote`: MANUAL/INVARIANT/INFERRED)** 가 더 강한 리스크 신호다. → **리스크 기반 분류로 사람의 주의를 위험군에 집중**.

## 3. 개선안

### D57 — 리스크 기반 검토 Triage (#4 해소) · 핵심

**⚠ 설계 보정 (사용자 지적)**: 초기안은 🔴을 `source=INFERRED`로 정의했으나, **참고문서(매뉴얼)가 없는 DOM-only 시험은 모든 TC가 INFERRED** → 🔴이 100%가 되어 무의미. 따라서 **참고문서와 무관하게 항상 변동하는 신호(생성 신뢰도·기법)를 주 기준**으로 삼고, source는 *있을 때만 보너스*로 쓴다.

**버킷 = 위험점수(risk score) 기반** (규칙 기반·LLM 불필요·임계값 `RunConfig` 노출):
```
risk_score = gen_confidence (0~1, 항상 존재·주축)
           + 0.15 if source=MANUAL   (참고문서 근거 — 없으면 미적용)
           + 0.10 if source=INVARIANT
           + 0.10 if technique=happy_path        (단순 = 저위험)
           − 0.05 if technique∈{negative_deep, cross_feature}  (추론 강함)
           − 0.05 if negative_category∈{injection_or_security, permission_denied}  (민감)
```
| 버킷 | 규칙 | 의미 |
|---|---|---|
| 🔴 **집중 검토** | `risk_score < 0.45` | 저신뢰·추론 강함 = 거짓 가능성↑ |
| 🟢 **안전(일괄승인 후보)** | `risk_score ≥ 0.75` | 고신뢰·근거 확실·단순 |
| 🟡 **빠른 확인** | 그 외 | 스폿 체크 |

**실측 검증 — 참고문서 유무 모두에서 동작**:
| 시나리오 | 🔴 집중 | 🟡 확인 | 🟢 일괄 |
|---|---|---|---|
| 참고문서 있음(source 반영) | 30% | 40% | 30% |
| **DOM-only(전부 INFERRED 가정)** | **34%** | 47% | 20% |

→ **DOM-only에서도 🔴이 ~34%로, 100% 검토가 아니다.** 신뢰도가 기법별로 갈리기 때문(happy_path 0.72 vs negative_deep 0.49). 근거 신호(기법별 평균 신뢰도)는 본 문서 측정으로 확인.

> 설계적 입장: DOM-only는 본질적으로 신뢰도가 낮아 🔴이 다소 큰 게 **정직한 결과**(위험을 인위적으로 숨기지 않음). 그래도 *우선순위*(🔴 먼저 + 🟡/🟢 샘플링)이지 *전수 검토*가 아니다. 임계값을 낮추면 더 공격적으로, 높이면 더 보수적으로 — 시험 정책에 맞춰 조정 가능.

**부담 감소 메커니즘**: 검토자는 ① 🟢 **원클릭 일괄 승인** → 23% 큐에서 제거, ② 🔴 **22%에 집중**(꼭 봐야 하는 set), ③ 🟡는 스폿 체크. ⇒ **"꼭 정독" 대상이 100% → ~22%**.

**UI**:
- 상단 **버킷 칩** "🔴 집중 N / 🟡 확인 N / 🟢 안전 N" — 클릭 시 필터.
- **필터·정렬**: 버킷·기법·상태·대분류·출처별. 기본 정렬 = **🔴 우선**.
- **버킷별 일괄 처리**: "🟢 전체 승인", "🔴만 보기" 등.
- **진행률을 위험군 기준으로**: "🔴 집중 검토 38건 중 12건 완료" → *끝이 보이는* 심리 효과(#4).
- 행/상세에 **"왜 검토하나" 한 줄** 자동 생성: 예 *"INFERRED 근거(매뉴얼 없음) + 신뢰도 0.38 → AI 추론, 검증 필요"*.

**구현 주의**: 정렬·필터 시 **행→TC 매핑**(현재 `self._tcs[row]`)이 깨지지 않게 TC를 직접 참조하는 모델 구조로(행에 tc_id/원본 인덱스 보관).

**meta.json**: `review_buckets = {red, yellow, green, thresholds}`.

### D58 — 상세 패널 강화 (#3 해소)

**재사용**: `failure_detail.py`의 `_resolve_screenshot`(run 교차검색)·전후이동·←/→ 키.

- **스크린샷 상시 표시**: TC의 `screenshot_file`(기능이 발견된 페이지) → 검토 맥락.
  - ⚠ **선행 보완**: 현재 V10 보완 TC는 screenshot_file 미상속(`stage3_verify._add_v10_tcs`가 leaf의 값을 복사 안 함) → 10%만 보유. **leaf→TC 전파 추가** 필요(D56과 함께).
- **전후 TC 이동** 버튼 + **키보드**: A=승인, E=수정, R=거부, ←/→=이전/다음. 484건을 한 손으로 훑기.
- **결정 근거 배지**: 출처(MANUAL/INVARIANT/INFERRED) + 신뢰도 + `applied_invariant`/`related_defect_id` 링크 표시.
- **정보 우선순위 재배치**: 스크린샷 → 시나리오/기대출력 → 근거 배지 → (보조) 분류·요구사항ID.

- **기존 기능 보존**: A/E/R/P·재생성·Excel·전체승인/거부 유지.
- **커버리지 무관**: triage는 *검토 동선*만 바꿈, TC 자체·실행 대상은 불변.
