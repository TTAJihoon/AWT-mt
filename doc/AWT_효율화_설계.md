## 1. 현재 실행 결과 — 무엇이 문제인가

그누보드5 실측 기준 (run `c38e532c`):

| 지표 | 값 | 의미 |
|---|---|---|
| 총 TC 수 | 104개 | Stage 4 게이트에서 38개 승인 |
| 통과 / 실패 | 34 / 4 | 실패 4건 전부 시나리오/자동화 한계 |
| 인증 도메인 TC 비율 | ~30% | 로그인 관련 TC가 과대표집 경향 |
| Reviewer Gate 대상 | 승인된 38개 | 104개 전체를 검토해야 하는 부담 |

**핵심 불편 2가지:**

1. **페이지마다 헤더 로그인 박스가 중복 추출** → 인증 기능이 TC를 비정상적으로 많이 차지
2. **Reviewer Gate에서 TC 전수 검토 압박** → 104개를 다 봐야 한다는 부담

## 2. 페이지 추출 정제 — 쓸데없는 중복 걸러내기

### 2-1. 문제: 전역 컴포넌트가 페이지마다 중복 추출

그누보드5의 헤더에는 로그인 박스(`#ol_id`, `#ol_pw`, `#ol_submit`)가 있다.  
BFS로 30개 페이지를 수집하면 이 박스가 **30번 추출**되어 기능 목록에 30개의 유사 "로그인" 기능이 생성된다.

```
[페이지 1] 헤더 로그인 박스 추출 → "로그인" leaf 생성
[페이지 2] 헤더 로그인 박스 추출 → "로그인" leaf 생성 (중복)
[페이지 3] 헤더 로그인 박스 추출 → "로그인" leaf 생성 (중복)
...
→ TC 설계 시 로그인 TC가 전체의 30% 이상 차지
```

### 2-2. 해결책: 전역 컴포넌트 dedup (D51) ✅ 구현 완료

**원리:** 동일한 DOM 요소 지문(tag + id + name + text)이 수집 페이지의 **40% 이상**에서 등장하면 "전역 컴포넌트"로 분류하여 `__global__` 가상 페이지에 **1번만** 모아서 처리한다.

```
[30개 페이지 수집]
  → 지문별 등장 횟수 집계
  → 로그인 박스: 44/89 페이지(49.4%) → 전역 판정(임계 40%)
  → __global__ 페이지에 1벌만 보존
  → 각 개별 페이지에서는 제거

결과: "로그인" leaf = 1개 (30개 → 1개)
```

**핵심 안전장치:**
- 전역 요소는 **삭제가 아닌 이동** → 기능 누락 없음
- 전역 비율 임계값(`global_ratio`)을 `RunConfig`에서 조정 가능
- 페이지 수가 5개 미만일 때는 비활성화 (오탐 방지)
- feature에 `scope: "global" | "page"` 필드로 구분 표시

**효과 (실측 추정):**
| | 이전 | 이후 |
|---|---|---|
| 인증 도메인 TC 비율 | ~30% | 목표 ~10%로 수렴 |
| DOM_SPEC LLM 호출 | 30회 | ~11회 (전역 1 + 고유 페이지 10) |

---

### 2-3. 해결책: 카테고리 통제 어휘 (D52) ✅ 구현 완료

**문제:** LLM이 같은 "로그인" 기능을 호출할 때마다 다른 이름을 붙인다.

```
User Management    ┐
Authentication     ├── 동일한 도메인, 다른 이름 → TC 분산
Account Management ┘
```

**해결:** 대분류를 **12개 고정 어휘**로 강제 통제한다.

```
회원·인증 / 게시판·콘텐츠 / 검색·필터 / 네비게이션·메뉴 / UI·접근성 /
결제·쇼핑 / 폼·입력검증 / 알림·고객지원 / 관리자 / 정보표시·정책 / 설정·환경 / 기타
```

**규칙:** LLM 프롬프트에 목록을 주입하여 "목록 외 대분류 생성 금지" 강제.  
`app/core/taxonomy.py`의 `coerce_major()` 함수가 생성 후에도 안전망 역할.

**효과 (그누보드5 2,939개 feature 실측):**
- 대분류 distinct: 436종 → **12종** (통제 어휘 97.7% 흡수)
- "User Management/Authentication/Account Management" → **"회원·인증" 단일 통합**

---

### 2-4. 해결책: 기능 확정 게이트 (D53) ✅ 구현 완료

**문제:** Stage 1 완료 후 Stage 2(TC 설계)로 곧바로 진입 → 사용자가 중간에 불필요 기능을 제거할 기회가 없음.

**해결:** Stage 1 완료 후, TC 설계 전에 **기능 확정 다이얼로그** 표시.

```
[Stage 1 완료]
    ↓
[기능 확정 게이트]
  대분류별 leaf 수 / 예상 TC 수 / 전체 비중(%) 표시
  과대 도메인 클릭 → 개별 leaf 제외 가능
  "확정" 클릭 → Stage 2 진행
    ↓
[Stage 2: 확정된 기능만 TC 설계]
```

**핵심:** 기본은 **무조작 통과** — 건드리지 않으면 기존과 동일. 의심스러운 기능이 있을 때만 조정.

## 3. TC 검토 효율화 — 검토 부담 100% → 22%로

### 3-1. 문제: "이 TC 다 봐야 하나요?"

현재 Stage 4 Reviewer Gate는 TC 전체를 일렬로 나열한다.  
104개 TC가 있다면 104개 모두 검토 대상처럼 보인다.

**실제 목적:** Reviewer Gate(D22)의 목적은 *모든 TC 검수*가 아니라  
**실행 전에 거짓·부실 TC를 걸러내는 것**이다.  
→ 위험한 TC에만 집중하고, 안전한 TC는 빠르게 통과시키면 된다.

---

### 3-2. 해결책: 위험점수 기반 3-버킷 분류 (D57) ✅ 구현 완료

TC마다 **위험 점수(risk_score)**를 자동 계산하여 3개 버킷으로 분류한다.

**위험점수 계산식:**
```
risk_score = gen_confidence               (0~1, 생성 신뢰도 — 항상 존재)
           + 0.15  if source = MANUAL     (매뉴얼 근거 있음)
           + 0.10  if source = INVARIANT  (불변 규칙 근거)
           + 0.10  if technique = happy_path        (단순 시나리오)
           − 0.05  if technique ∈ {negative_deep, cross_feature}  (추론 강함)
           − 0.05  if negative_category ∈ {injection_or_security, permission_denied}
```

**버킷 분류:**

| 버킷 | 조건 | 의미 | 권장 행동 |
|---|---|---|---|
| 🔴 **집중 검토** | risk_score < 0.45 | 저신뢰·추론 강함 · 거짓 가능성↑ | 반드시 정독 |
| 🟡 **빠른 확인** | 0.45 ≤ score < 0.75 | 중간 신뢰도 | 스폿 체크 |
| 🟢 **안전** | risk_score ≥ 0.75 | 고신뢰·근거 확실·단순 시나리오 | 일괄 승인 |

**실측 분포 (참고문서 있는 경우):**

| 버킷 | 비율 | 104개 기준 |
|---|---|---|
| 🔴 집중 검토 | 약 30% | ~31개만 정독 |
| 🟡 빠른 확인 | 약 40% | 샘플링 |
| 🟢 안전 (일괄 승인) | 약 30% | 클릭 1번으로 처리 |

→ **"꼭 정독" 대상이 100% → 약 30%로 감소**

**DOM-only 시험(매뉴얼 없는 경우)에서도 동작:**  
매뉴얼 없으면 모든 TC가 INFERRED → 종전 방식으로는 🔴 100%.  
D57은 신뢰도·기법 기반이므로 DOM-only에서도 🔴 약 34%로 수렴.

### 3-3. Reviewer Gate UI 개선 (D58) ✅ 구현 완료

**현재 화면에서 확인 가능한 기능:**

| 기능 | 설명 |
|---|---|
| 버킷 칩 표시 | "🔴 집중 N / 🟡 확인 N / 🟢 안전 N" — 클릭 시 해당 버킷만 필터 |
| 🟢 일괄 승인 버튼 | 안전 버킷 전체를 클릭 한 번으로 승인 |
| 버킷 기준 진행률 | "🔴 31건 중 12건 완료" — 끝이 보이는 심리적 효과 |
| 키보드 단축키 | A=승인, E=수정, R=거부, ←/→=이전/다음 TC |
| 상세 패널 | 스크린샷 + 근거 배지(MANUAL/INVARIANT/INFERRED + 신뢰도) |
| "왜 검토하나" 한 줄 | 예: "INFERRED 근거 + 신뢰도 0.38 → AI 추론, 검증 필요" |
| 기본 정렬 | 🔴 우선 → 가장 위험한 TC부터 보여줌 |

### 3-4. 검토 흐름 비교

**개선 전:**
```
104개 TC → 일렬 나열 → 처음부터 끝까지 전수 검토
시간 소요: (검토자 판단에 따라) 수십 분~수 시간
```

**개선 후:**
```
104개 TC → 위험점수 자동 계산 → 3-버킷 분류
  1단계: 🟢 30개 → "일괄 승인" 클릭 1회 (30초)
  2단계: 🔴 31개 → 정독 (가장 위험한 것부터, 끝이 보임)
  3단계: 🟡 43개 → 스폿 체크 (필요한 것만)
총 검토 필요 TC: ~31개 (전체의 30%)
```

## 4. 성능 개선 — 생성 시간 단축

### 4-1. 그룹핑으로 LLM 호출 90% 감소 (D54) ✅ 구현 완료

**이전:** leaf 1개마다 TC_DESIGN LLM을 1회 호출  
→ 26개 leaf = 26회 호출 (각 leaf가 고립되어 기능 관계 파악 불가)

**이후:** 같은 URL 페이지의 leaf를 묶어서 1회 호출 (페이지 그룹)  
→ 26개 leaf → 3개 그룹 = 3회 호출

```
그룹 1: 게시판 관련 12개 기능 → LLM이 관계를 보고 cross-feature TC 자동 설계
그룹 2: 회원 관련 12개 기능 → LLM이 로그인-글쓰기 시나리오 파악
그룹 3: 관리자 관련 2개 기능 → 빠른 처리
```

**추가 효과:** LLM이 같은 페이지 기능들의 관계를 보므로  
교차 시나리오 TC(`cross_feature`)가 자연스럽게 생성된다.

### 4-2. 병렬 실행 (D55) ✅ 구현 완료

독립적인 LLM 호출을 동시에 실행 (기본 동시성 6).

```
이전: 그룹 3개를 순차 실행 → 총 시간 = 3 × (응답시간)
이후: 그룹 3개를 동시 실행 → 총 시간 ≈ 1 × (응답시간)
```

**결정성 유지:** 결과는 항상 입력 순서로 재정렬 → TC ID 안정.

## 5. 종합 장점 요약

### 5-1. 페이지 추출 정제 효과

| 문제 | 해결책 | 결정 | 상태 |
|---|---|---|---|
| 헤더/푸터 요소가 페이지마다 중복 추출 | 전역 컴포넌트 dedup — 40%↑ 등장 요소를 1벌만 처리 | D51 | ✅ |
| 인증 도메인이 3개 이름으로 분열 | 대분류 12종 고정 어휘 강제 | D52 | ✅ |
| 불필요 기능이 TC를 증식시킴 | Stage 1~2 사이 기능 확정 게이트 | D53 | ✅ |

**수치 효과:**
- 대분류 distinct 436종 → 12종 (97.7% 흡수)
- 인증 도메인 TC 비율: ~30% → ~10% 목표
- LLM 호출 횟수 최대 40% 감소

### 5-2. TC 검토 효율화 효과

| 문제 | 해결책 | 결정 | 상태 |
|---|---|---|---|
| TC 전수 검토 압박 | 위험점수 3-버킷 자동 분류 | D57 | ✅ |
| 검토에 필요한 맥락 정보 부족 | 스크린샷·근거 배지·키보드 탐색 | D58 | ✅ |

**수치 효과:**
- "꼭 정독" 대상: 100% → **약 30%**
- 🟢 버킷 일괄 승인으로 30%는 클릭 1번 처리
- 키보드(A/E/R/←/→)로 수백 건도 한 손으로 처리 가능

### 5-3. 성능 개선 효과

| 문제 | 해결책 | 결정 | 상태 |
|---|---|---|---|
| Leaf 1개씩 순차 설계로 LLM 호출 多 | 페이지 그룹 단위 설계 | D54 | ✅ |
| 모든 LLM 호출이 순차 실행 | ThreadPoolExecutor 병렬화 (기본 동시성 6) | D55 | ✅ |
| V10 보완 호출이 leaf마다 순차 | 배치 그룹 처리 | D56 | ✅ |

**수치 효과:**
- Stage 2 LLM 호출: leaf 개수 × N → 그룹 수 × N (최대 90% 감소)
- 동시성 6 적용 시 직렬 등가 시간 추가 6분의 1

### 5-4. 기타 구조적 장점

| 항목 | 내용 |
|---|---|
| **손실 0 원칙** | 전역 dedup은 삭제가 아닌 이동. 기능 확정 게이트 제외만 명시적 삭제 |
| **옵트아웃 가능** | 모든 개선 기능은 RunConfig 플래그로 비활성화 가능 — 회귀 없음 |
| **결정성** | 버킷 분류·전역 탐지 모두 규칙 기반(LLM 불필요) → 재현 가능 |
| **추적성** | 모든 변경 사항을 meta.json에 기록 → 재현·감사 가능 |
| **단계 재개** | tc_verified.json 있으면 Stage 4부터 이어서 진행 — 처음부터 재시작 불필요 |
