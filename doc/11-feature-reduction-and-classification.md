## 3. Q3 답 — "TC 2종류만?" → **오해**

`대분류`(제품 도메인 축)와 `기법`(시험 설계 축)은 **직교한다**.

- 커버리지를 결정하는 건 `design_technique` → 실측 **7종 모두 생성**(정상·경계·등가·기본부정·심층부정·상태전이·교차기능).
- 화면에서 "2종류"로 보인 건 **대분류 컬럼**이며, 이는 분류 버그(Q2)일 뿐 TC 종류가 아니다.

> 결론: 커버리지는 다양하다. TC 목록의 `기법` 컬럼에서 7종 확인 가능.

## 4. Q2 답 — 대분류 붕괴: 근본 원인 2개 + 수정(Fix B·C)

### 근본 원인 A — 매뉴얼 헤딩 레벨 off-by-one
매뉴얼 구조:
```
# 그누보드5 기능 명세서 (AWT Stage 1 입력용)   ← 최상위 #이 "문서 제목" 1개뿐
## 1. 회원 관리                                ← 실제 제품 도메인이 ## 레벨
### 1.1 회원가입                               ← 기능이 ### 레벨
```
`_extract_leaves_from_text`는 `#`→대분류로 가정 → **26개 leaf 전부 대분류 = 문서 제목**.
진짜 도메인(회원 관리·게시판...)은 중분류로 밀려남.

### 근본 원인 B — 문서 제목이 통제 어휘에 오염
제목의 "...Stage 1 **입력**용"에 `입력` 포함 → `coerce_major`가 `폼·입력검증` 규칙에
매칭 → 26개 전부 폼·입력검증. (로그 "대분류 통제 어휘 보정 — 26개 정규화"가 이것)

### 수정 — Fix B (헤딩 자동 보정)
`app/core/stage1_ingest.py` `_extract_leaves_from_text`:
- `#`이 1개뿐이고 `##`이 2개 이상이면 **레벨 한 칸 승격**: `##`→대분류, `###`→소분류.
- 일반 다중 `#` 매뉴얼은 기존 매핑 유지(회귀 없음).

### 수정 — Fix C (메타 문자열 차단)
`app/core/taxonomy.py` `coerce_major`:
- "명세서·기능 명세·specification·입력용·참고문서" 등 **문서 메타 토큰**이 들어간
  문자열은 키워드 매칭 전에 `기타`로 분류(오매칭 방지). 보수적 토큰만 선정.

### 검증 결과 (매뉴얼 26 leaf 재적용)
| 대분류(보정 후) | 개수 |
|---|---|
| 게시판·콘텐츠 | 7 |
| 관리자 | 5 |
| 회원·인증 | 4 |
| 결제·쇼핑 | 3 |
| 6. 접근 제어 및 보안 | 3 (키워드 미매칭 → 원본 유지) |
| 8. 경계 조건 및 예외 | 3 (원본 유지) |
| 검색·필터 | 1 |

→ `대분류=회원·인증 | 중분류=1. 회원 관리 | 소분류=1.1 회원가입` 로 정정.
(이전: 98개 전부 폼·입력검증)

> **남은 꼬리**: 섹션 6(접근 제어 및 보안)·8(경계 조건)은 통제 어휘 키워드 미매칭으로
> 원본 섹션명 유지. 폼·입력검증보다 정직하나, 추후 한글 키워드(권한·보안) 보강 가능.

## 5. Q1 — 4인 전문가 토론: 기능 1082개 축소

> 패널: **테스트 아키텍트(TA)** · **SW 엔지니어(SE)** · **QA 리드(QA)** · **UX/PM**
> 데이터: run B 1082개 중 `네비게이션·메뉴` 257개(23.8%)가 최대 비중.

### 라운드 1 — 원인 진단
- **TA**: 257개가 네비게이션·메뉴 = 거의 "○○ 메뉴 링크 클릭→이동". 시험 가치상
  서로 다른 기능이 아니라 **"네비게이션 이동" 1개 기능의 인스턴스 257개**. 링크마다
  TC를 만들 이유가 없다.
- **SE**: D51 전역 dedup 시그니처가 `text` 포함이라 메뉴 텍스트가 다르면 안 묶임.
  40% 등장 기준도 카테고리별 메뉴는 못 넘김.
- **QA**: Stage 1b LLM 통합의 구멍 ① `consolidate_features` off면 미동작 ②
  배치가 입력순 250개라 같은 메뉴가 다른 배치로 흩어지면 못 묶음.
- **UX/PM**: 1082개를 게이트에서 보는 것·257개 일일이 해제하는 것 둘 다 부담.

### 라운드 2 — 핵심 통찰 (Q1·Q2 같은 뿌리)
- **TA**: taxonomy가 **두 직교 축을 혼재**한다.
  - 제품 도메인: 회원·인증 / 게시판 / 검색 / 결제 / 관리자
  - 상호작용 유형: 폼·입력검증 / UI·접근성 / 네비게이션·메뉴
  회원가입은 *도메인=회원·인증*이며 동시에 *유형=폼*. LLM이 유형을 고르면
  ① 회원가입→폼·입력검증(Q2) ② 메뉴 링크→네비게이션 도메인 257개 폭증(Q1).
  **두 문제가 한 뿌리.**
- **QA**: 도메인 축만 남기면 "네비게이션 자체 시험"이 누락될 수 있으니 **삭제가 아닌
  대표 축약**이어야.
- **UX/PM**: 자동 축약 시 **추적성(meta.json 기록)** 필수 — 시험소 감사 방어.

**아이디어 A 구현 상세 (2축 분리):**
- taxonomy 12종을 **도메인 8종**(회원·인증/게시판·콘텐츠/검색·필터/결제·쇼핑/관리자/
  알림·고객지원/정보표시·정책/설정·환경)과 **상호작용 3종**(폼·입력검증/UI·접근성/
  네비게이션·메뉴)으로 분리.
- `classify_major(name, mid, leaf)`: leaf·중분류까지 보아 **도메인 우선** 분류.
  도메인 신호가 없을 때만 상호작용 유형 사용. → "폼·입력검증" major여도 leaf가
  "회원가입"이면 회원·인증으로 구제. 메뉴 링크는 가리키는 도메인으로 귀속.
- `coerce_major`는 하위호환 래퍼로 유지(`classify_major(name)`).
- 호출처: `stage1_ingest._refine_leaves`(DOM 경로) + `orchestrator` 통합 후 sweep.
- 회귀: 단위 검증 + pytest 112 passed(기존 실패 1건 무관).

**아이디어 4 구현 상세 (게이트 도메인 예산):** `app/ui/feature_gate.py`
- 기능 확정 게이트(D53)에 도메인(대분류)별 **TC 예산 상한 스핀박스** 추가.
- 기본값 = 전체(상한 없음) → 회귀 없음. 낮추면 해당 도메인을 상한만큼 제한.
- **대표 우선 선정**: 중분류 라운드로빈(커버리지 우선) + confidence 높은 순 →
  예산이 한 중분류에 쏠리지 않고 도메인 전반을 대표. 결정적(재현성).
- 게이트 확정 시 `kept_leaves`를 예산만큼 트리밍 → Stage 2는 자동으로 예산 준수
  (orchestrator/stage2 수정 불필요). `feature_gate.domain_budgets`로 추적성 기록.

**남은 후보(🔵):** 아이디어 3(전역 dedup 시그니처 text 제외), 5(noise 패턴 확장) — 필요 시 추가.
