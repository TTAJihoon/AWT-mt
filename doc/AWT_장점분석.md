## 1. 입력 자동화 — 기능리스트를 직접 만들지 않아도 된다

### 기본 접근법의 한계

기본 접근법에서 TC를 생성하려면 사전에 **기능리스트**를 사람이 수작업으로 작성해야 한다.  
실제 제품은 수십~수백 개의 화면과 기능을 가지며, 기능리스트 작성 자체가 상당한 공수를 요구한다.  
매뉴얼이 없는 제품이라면 TC 생성조차 시작하기 어렵다.

### AWT의 접근

**Stage 0 (DOM 스캔)** 이 이 문제를 해결한다.

- Playwright가 대상 URL에 접속해 BFS 방식으로 모든 페이지를 자동 크롤링
- 각 페이지의 DOM 요소(버튼, 폼, 링크, 메뉴 등)를 추출해 50개 단위 청크로 분할
- LLM이 DOM 청크를 읽고 `기능명(소분류) → 기능 명세(implicit_spec)` 형태로 기능을 자동 추출
- 매뉴얼 첨부 없이도 웹사이트 자체에서 기능리스트를 자동 생성
- 각 페이지별 스크린샷을 저장해 사용자가 제품 형상을 시각적으로 확인 가능

**결론**: 기능리스트 작성 공수를 제거하고, 누락 가능성도 줄인다.  
매뉴얼 없이 URL만 있으면 TC 생성까지 진행할 수 있다.

## 2. 설계 품질 — 7가지 기법 + 음성 커버리지 강제

### 기본 접근법의 한계

사람이 Claude에게 "TC를 만들어줘"라고 요청하면, 결과는 요청자가 어떻게 프롬프트를 작성하느냐에 전적으로 달려 있다.

- 별도로 명시하지 않으면 happy path(정상 흐름)만 생성되는 경향이 있다
- 경계값, 권한 오류, SQL injection 시나리오 등 **음성(negative) 케이스**가 누락되기 쉽다
- 매번 다른 수준의 TC가 나오며, 설계 기법의 적용이 일관되지 않는다

### AWT의 접근

**Stage 2 (TC 설계)** 는 각 기능(leaf)별로 아래를 체계적으로 강제한다.

#### 7가지 설계 기법 적용

| 기법 | 설명 |
|------|------|
| `happy_path` | 정상 흐름 (모든 leaf 필수) |
| `equivalence` | 등가 분할 — 입력 도메인별 대표값 |
| `boundary` | 경계값 — min/max ±1, 빈 입력, 길이 상한 |
| `negative_basic` | 기본 부정 — 명세 위반 입력에 대한 에러 처리 |
| `negative_deep` | 심층 부정 — 권한 우회, XSS, 메시지 적절성 |
| `state_transition` | 상태 전이 — 다단계 워크플로 step 누락·역순 |
| `cross_feature` | 기능 결합 — 의미 연관 기능과의 조합 |

#### 음성 카테고리 강제 (D49)

기능 유형에 따라 아래 5개 카테고리 중 적용 가능한 것은 반드시 TC에 포함되어야 한다:

| 카테고리 | 설명 |
|----------|------|
| `validation_failure` | 입력 형식·필수값 위반 |
| `duplicate_or_conflict` | 중복·동시성·충돌 |
| `permission_denied` | 권한 거부·만료 토큰 |
| `boundary_violation` | 경계값 초과 |
| `injection_or_security` | SQL injection, XSS, CSRF 등 보안 공격 |

#### 도메인 지식 주입

- **Domain Invariants**: 제품 유형별로 공통 적용되는 업계 불변 규칙 자동 주입
- **유사 결함 패턴**: 과거 유사 제품에서 발견된 결함 사례를 RAG로 검색해 LLM에 제공

**결론**: 사람이 직접 요청하지 않아도 시험 기법이 체계적으로 적용된다.  
특히 음성 케이스와 보안 케이스가 구조적으로 보장된다.

## 3. 자동 품질 검증 — V1~V10 루프

### 기본 접근법의 한계

Claude가 생성한 TC는 사람이 직접 읽으면서 품질을 검수해야 한다.  
TC가 100개를 넘으면 검수 자체가 상당한 부담이 되며,  
구조적 오류(시나리오 불명확, expected 누락 등)가 통과되기 쉽다.

### AWT의 접근

**Stage 3 (V1~V10 검증)** 이 생성 직후 자동으로 품질을 검사한다.  
현재 **6종(V1~V6, V10)** 이 구현되어 있으며, V7~V9는 Phase 2 예정이다.

#### V1 — 필수 필드 존재 + tc_id 형식

```
검사 대상: tc_id, 대분류, 중분류, 소분류, scenario, precondition,
           expected, requirement_id, design_technique, source_quote,
           gen_confidence — 11개 필드 전부 비어있지 않은지
           + tc_id가 "TC-001-003" 형식(TC-\d{3}-\d{3})인지
실패 시  : 해당 TC를 TC_REGEN(재생성) 대상으로 표시
```

LLM이 필드를 빠뜨리거나 tc_id 형식을 잘못 생성하는 경우를 잡아낸다.

#### V2 — source_quote 매뉴얼 대조

```
검사 대상: source_quote가 "MANUAL:"로 시작하는 TC에 한해서만,
           해당 인용문이 실제 매뉴얼 텍스트 안에 포함되어 있는지
예외      : "INFERRED:..." 또는 "INVARIANT:..." 출처는 검사 제외
실패 시  : 해당 TC를 TC_REGEN 대상으로 표시
```

LLM이 매뉴얼에 없는 내용을 마치 매뉴얼 인용인 것처럼 꾸며 쓰는 경우를 잡아낸다.  
"INFERRED:"나 "INVARIANT:"는 매뉴얼 원문 대조 대상이 아니므로 제외한다.

#### V3 — INFERRED 비율 임계

```
검사 대상: 전체 TC 중 source_quote가 "INFERRED:..."인 TC의 비율
임계값   : 30% 초과 시 실패
실패 시  : INFERRED TC를 gen_confidence 낮은 순으로 최대 10개씩
           TC_REGEN 대상으로 선별 → 매뉴얼 근거를 더 찾도록 유도
```

매뉴얼 근거 없이 추론에 의존한 TC가 너무 많으면 TC 신뢰도 전체가 낮아진다.  
일정 비율 이상은 매뉴얼 근거를 확보하도록 강제한다.

#### V4 — happy_path 과다 비율

```
검사 대상: 전체 TC 중 design_technique == "happy_path"인 TC의 비율
임계값   : 50% 초과 시 실패
실패 시  : happy_path TC 후반부 절반을 TC_REGEN 대상으로 선택
           → boundary, negative_deep 등 다른 기법으로 재생성 유도
```

기본 접근법의 가장 큰 문제인 "happy path만 잔뜩 나오는 현상"을 수치로 제어한다.

#### V5 — leaf 전체 커버리지

```
검사 대상: Stage 1에서 추출한 leaf(기능 목록) 전체에 대해
           각각 최소 1개 이상의 TC가 존재하는지
실패 시  : 미커버 leaf에 대해 전체 TC를 재생성 대상으로 표시
```

TC 생성이 특정 기능에만 집중되고 나머지 기능이 빠지는 경우를 방지한다.

#### V6 — Playwright 셀렉터 안정성 (Stage 5 시점)

```
실행 위치: Stage 5 자동 실행 완료 후 (app/validation/v6_selector_stability.py)
검사 내용: TC 실행에 사용된 Playwright 셀렉터(CSS/XPath)의 안정성
  - selector_stability : 셀렉터가 얼마나 고유하고 깨지기 어려운 형태인지
  - oracle_clarity     : 기댓값 검증 기준이 얼마나 명확한지
  - exec_confidence    : 위 두 값을 합산한 실행 신뢰도
출력      : TC에 exec_confidence 필드 추가 + failure_category 사전 분류
```

실행 단계에서 TC 품질을 다시 평가한다.  
셀렉터가 취약하면 실패 원인이 "제품 결함"이 아닌 "TC 작성 문제"임을 사전에 파악할 수 있다.

#### V10 — negative_category 커버리지 강제 (D49)

```
검사 대상: 각 leaf의 유형(로그인·업로드·결제 등)에 따라 적용 가능한
           음성 카테고리를 먼저 결정하고, 실제 TC가 그 중 60% 이상을 커버하는지
           + 커버된 각 카테고리당 TC가 1개 이상인지

예시      : "로그인" leaf → 적용 카테고리: validation_failure,
                                          duplicate_or_conflict,
                                          boundary_violation
            → 이 중 2개 이상 카테고리에 TC가 각각 1개 이상 있어야 통과

실패 시  : V1~V5와 달리 기존 TC를 재생성하지 않고, 누락 카테고리를 타깃으로
           TC_DESIGN을 재호출해 새 TC를 추가 (기존 TC 보존)
```

> **V10이 V1~V5와 다른 이유**: V1~V5는 기존 TC를 교체(rewrite)하지만,  
> V10은 기존 TC는 건드리지 않고 **누락 카테고리 TC를 신규 추가**한다.  
> 기존 TC를 교체하면 이미 통과한 다른 검증이 깨질 수 있기 때문이다.

#### V7~V9 — Phase 2 예정

현재 코드에 구현체가 없는 예약 번호다.  
Phase 2에서 화면 간 일관성(V7), 성능 기댓값(V8), 접근성 기준(V9) 등이 추가될 예정이다.

#### 검증 전체 루프 흐름

```
Stage 2에서 TC 생성
       ↓
V1, V2, V3, V4, V5, V10 동시 검사
       ↓
┌── 구조적 실패(V1~V5)가 있으면 ──→ TC_REGEN 호출 ──→ 최대 3회 반복
│
└── V10 누락이 있으면 ──→ TC_DESIGN 재호출로 TC 추가 (병행)
       ↓
3회 초과해도 실패 → 해당 TC에 "INFERRED: max_retry_exceeded" 마킹
                   + review_status = pending (Stage 4 검토 대상으로 분류)
       ↓
Stage 5 실행 완료 후 V6 셀렉터 안정성 검사 → exec_confidence 기록
```

**결론**: 생성 → 검증 → 재생성 루프가 내장되어 있어  
사람이 일일이 구조적 오류를 찾아내지 않아도 된다.  
3회 재시도 후에도 통과 못 한 TC만 사람 검토 대상으로 명시적으로 표시된다.

## 4. 재현성과 일관성 — 동일한 입력이면 동일한 구조 보장

### 기본 접근법의 한계

Claude에게 같은 내용을 두 번 요청해도 결과가 다르다.  
오늘 생성한 TC와 내일 생성한 TC의 형식이 다를 수 있고,  
누가 요청하느냐에 따라 TC 수와 깊이가 달라진다.

### AWT의 접근

- **run_id** 기반 실행 관리: 모든 실행에 고유 ID가 부여되어 입력·중간 결과·산출물이 `data/runs/{run_id}/` 에 저장됨
- **Call Contract**: LLM 호출마다 고정된 스키마(입력 필드, 출력 필드, JSON 형식)가 있어 응답 형식이 강제됨
- **tc_id 정규화**: LLM이 형식을 틀려도 `TC-{leaf:03d}-{tc:03d}` 형식으로 강제 덮어씀
- **Stage 재개**: `tc_verified.json` 이 저장되어 있으면 Stage 4부터 다시 시작 가능 (`load_from_stage3`)

#### Stage 재개 상세 — load_from_stage3

각 Stage 완료 시마다 중간 결과가 파일로 저장된다.

```
data/runs/{run_id}/
├── feature-spec-draft.json   ← Stage 0 결과 (기능 목록)
├── screenshots/              ← Stage 0 스크린샷 모음
├── ingest/
│   └── manual.txt            ← Stage 1 결과 (파싱된 매뉴얼)
├── tc_raw.json               ← Stage 2 결과 (검증 전 TC)
├── tc_verified.json          ← Stage 3 결과 (검증 완료 TC) ★ 재개 기준점
├── tc_gated.json             ← Stage 4 결과 (검토 결정 반영)
├── tc_executed.json          ← Stage 5 결과 (실행 결과 포함)
├── tc_review.xlsx            ← Stage 3→4 검토용 Excel
└── tc_final.xlsx             ← Stage 7 최종 산출물
```

`load_from_stage3(run_id)` 호출 시 아래 3단계가 자동으로 실행된다.

1. `tc_verified.json` → `self.tcs` 복원
2. `ingest/manual.txt` → `self.ingest_result` (매뉴얼 텍스트 + leaf 목록) 복원
3. `self._stage = 3` 으로 상태 설정 → 이후 Stage 4부터 정상 진행

**Stage 재개가 필요한 실제 상황**

| 상황 | 재개 방법 |
|------|----------|
| Stage 5 실행 도중 오류 종료 | `tc_gated.json`이 있으면 Stage 5만 재실행 |
| 검토 결정(A/E/R)을 바꾸고 싶다 | `tc_verified.json`에서 Stage 4 다시 진행 |
| 다른 PC에서 동일 제품 재실행 | run 폴더째 복사 후 `load_from_stage3(run_id)` |
| Stage 0~3 완료 후 며칠 뒤 실행 재개 | 대시보드에서 해당 run 더블클릭 → 자동 재개 |

**결론**: "어떻게 프롬프트를 쓰느냐"에 관계없이  
코드 수준에서 TC 구조가 표준화된다.  
또한 Stage 단위 저장으로 중간 실패 시 처음부터 다시 하지 않아도 된다.

## 5. 규모 확장성 — 컨텍스트 윈도우 한계 초월

### 기본 접근법의 한계

Claude의 컨텍스트 윈도우 안에 들어오는 기능만 처리할 수 있다.  
제품 규모가 크면(수십 페이지, 수백 기능):

- 기능리스트를 잘라서 여러 번 요청해야 한다
- 여러 번 요청하면 tc_id 충돌, 중복 TC, 일관성 깨짐 문제가 발생한다
- 대화 맥락이 길어지면서 앞의 내용을 잊어버린다

### AWT의 접근

- **leaf 단위 분할 처리**: Stage 2는 기능리스트를 leaf 단위로 쪼개 각각 독립적인 LLM 호출로 처리
- **DOM 청킹**: Stage 0에서 DOM 요소를 50개 단위로 분할해 12,000자 트런케이션 문제 원천 차단
- **tc_id 자동 관리**: leaf 번호 × tc 번호로 전역 유일 ID가 보장됨
- **중간 결과 저장**: 각 Stage 완료 후 JSON으로 저장하므로 중간에 실패해도 이어서 실행 가능

**결론**: 200개 기능, 500개 TC 규모도 컨텍스트 한계 없이 일관되게 처리한다.

---

## 6. Human-in-the-loop — 검수 워크플로가 파이프라인 안에 있다

### 기본 접근법의 한계

Claude가 생성한 TC를 검토하는 프로세스가 별도로 존재하지 않는다.  
검토자가 Excel을 직접 수정하고, 어떤 TC를 왜 수정했는지 기록이 남지 않는다.

### AWT의 접근

**Stage 4 (Reviewer Gate)** 가 검수 워크플로를 내장한다.

- Stage 3 완료 후 `tc_review.xlsx` 자동 생성 — 드롭다운 상태 선택 포함
- 앱 내 검토 UI: TC 목록 테이블 + 더블클릭 상세 팝업
- 각 TC마다 **A/E/R/P (approved / edited / rejected / pending)** 상태 지정
- `reviewer_id` + `reviewer_note` 필드로 누가 왜 수정했는지 기록
- approved/edited만 Stage 5(실행)로 진행, rejected는 제외

**결론**: 사람의 검수가 파이프라인의 공식 단계로 포함되어 있어  
"생성 → 무조건 실행"이 아닌 "생성 → 검수 → 실행"의 품질 게이트가 자동 적용된다.

---

## 7. 실행 자동화 — TC가 바로 실행된다

### 기본 접근법의 한계

Claude가 생성한 TC는 문서(텍스트)다.  
실제 웹 제품에 대해 실행하려면:

1. 사람이 TC를 읽고 수동으로 브라우저를 조작해야 한다
2. 또는 별도 자동화 도구(Selenium, Playwright 등)에 TC를 다시 변환해야 한다

이 두 단계 사이에는 언제나 사람의 추가 공수가 요구된다.

### AWT의 접근

**Stage 5 (Playwright 실행)** 이 TC를 직접 실행한다.

- Stage 4에서 approved/edited된 TC가 Playwright 스크립트로 변환되어 실행
- 인증이 필요한 제품은 `auth_sequence`(로그인 절차)를 사전 정의하면 자동 처리
- 실행 결과(`actual`, `result`, `exec_confidence`)가 TC에 직접 기록됨
- 실행 후 `tc_executed.json` 저장으로 재실행 없이 Stage 6~7 진행 가능

**결론**: "TC 문서 → 실행 가능한 테스트"로의 변환 과정이 자동화되어  
수동 테스트 또는 별도 자동화 구축 없이 실행 결과를 얻는다.

---

## 8. 실패 원인 분석 — pass/fail 너머의 정보

### 기본 접근법의 한계

기본 접근법에서 TC를 직접 실행해 fail이 나오면,  
"왜 실패했는가"를 분석하는 것은 전적으로 사람의 몫이다.  
실패가 다음 중 어떤 원인인지 분류하는 데도 시간이 걸린다:

- 실제 제품 결함인가 (real defect)
- TC 자체의 시나리오 오류인가 (scenario_error)
- 셀렉터가 깨진 것인가 (selector_broken)
- 기댓값 설정이 잘못된 것인가 (expected_mismatch)

### AWT의 접근

**Stage 6 (실패 원인 분석)** 이 이 작업을 자동화한다.

#### 5가지 failure_category (D50)

| 카테고리 | 의미 |
|---------|------|
| `selector_broken` | Playwright 셀렉터가 DOM 변경으로 깨짐 — TC 수정 필요 |
| `scenario_error` | 시나리오 자체의 오류 — TC 재설계 필요 |
| `expected_mismatch` | 기댓값과 실제값의 불일치 — TC 수정 또는 제품 확인 |
| `real_defect` | 실제 제품 결함 — 개발팀 보고 대상 |
| `fictional_positive` | LLM 생성 TC가 실제와 다른 가상 시나리오 — TC 폐기 |

- V6 정적 분석 (셀렉터 안정성)을 먼저 수행해 명백한 분류는 LLM 호출 없이 처리
- 복잡한 케이스만 LLM 동적 분석으로 보강 (비용 최적화)

**결론**: "몇 개 실패"가 아니라 "왜 실패했고 어떻게 처리해야 하는가"까지 자동 분류된다.

## 9. 표준 기반 산출물 — 인증기관 제출 가능한 형식

### 기본 접근법의 한계

Claude가 생성한 TC의 형식은 프롬프트에 따라 매번 다르다.  
ISO/IEC 25023 요구사항을 충족하는 형식으로 일관되게 나오지 않으며,  
별도로 형식을 맞추는 작업이 추가로 필요하다.

### AWT의 접근

**Stage 7 (Excel 산출)** 이 표준화된 형식으로 최종 결과를 생성한다.

- `표준 양식` 시트: ISO/IEC 25023 기반 컬럼 구성 (tc_id, 대/중/소분류, scenario, precondition, expected, actual, result, failure_reason, screenshot_file)
- `AWT_Meta` 시트: 설계 근거 (requirement_id, design_technique, source_quote, gen_confidence, exec_confidence, review_status, reviewer_id)
- 결과 셀 색상 자동 적용: pass=연두, fail=빨강, blocked=노랑
- gen_confidence 수준별 색상: 0.85 이상=연두, 0.50~0.85=노랑, 이하=빨강
- Stage 0 기능목록도 별도 Excel로 다운로드 가능

**결론**: 생성 직후 인증기관 제출 또는 고객사 공유에 바로 사용할 수 있는  
일관된 형식의 문서가 만들어진다.

## 10. 추적성 — TC와 근거를 연결한다

### 기본 접근법의 한계

Claude가 TC를 만들면 "어떤 요구사항에서 나온 TC인가", "매뉴얼의 어떤 항목을 근거로 했는가"가 명시되지 않는 경우가 많다.  
추후 감사(audit)나 재검토 시 TC의 근거를 역추적하기 어렵다.

### AWT의 접근

모든 TC에 다층적 추적 정보가 내장된다.

| 필드 | 연결 대상 |
|------|----------|
| `requirement_id` | 기능리스트의 대>중>소 경로 — 어떤 기능에서 나온 TC인지 |
| `source_quote` | 매뉴얼 원문 발췌 — 어떤 명세를 근거로 했는지 |
| `design_technique` | 어떤 설계 기법을 적용했는지 |
| `screenshot_file` | Stage 0에서 찍은 어떤 화면의 기능인지 |
| `gen_confidence` | LLM이 TC를 생성할 때의 신뢰도 수치 |
| `review_status` / `reviewer_id` | 누가 검토했고 어떤 결정을 내렸는지 |

**결론**: TC 1개에서 출발해 요구사항 → 매뉴얼 원문 → 화면 → 검토자 결정까지  
역방향으로 완전히 추적할 수 있다.

## 11. 도메인 지식 내재화 — 경험이 코드로 축적된다

### 기본 접근법의 한계

사람이 Claude에게 요청할 때마다 시험 전문가의 노하우를 프롬프트에 담아야 한다.  
경험이 담당자 개인에게 귀속되며, 담당자가 바뀌면 품질이 떨어진다.

### AWT의 접근

시험 경험이 코드와 자산(assets)으로 축적된다.

#### 자산 구조

```
data/assets/
├── defect-catalog/           ← 발견된 결함 + AI 생성 패턴 제안
│   ├── BOARD_CMS/            ← 제품 유형별 분리
│   │   ├── DEF-2026-BRD-001.json   예) "수정·삭제 소유권 검증 없음"
│   │   └── DEF-2026-BRD-002.json
│   └── USER_AUTH/
└── domain-invariants/        ← 제품 유형별 불변 규칙 (YAML)
    ├── BOARD_CMS.yaml         예) "비로그인 수정·삭제 요청은 서버에서 거부"
    └── USER_AUTH.yaml         예) "빈 비밀번호 로그인 시도는 거부해야 한다"
```

#### 자산 생명주기

```
시험 중 결함 발견
  → 테스터가 사실(제목/현상/기대) 기록
  → PATTERN_EXTRACT LLM Contract가 patternProposal 자동 생성
  → 테스터/리드가 승인 or 기각
  → 승인된 패턴 → domain-invariants.yaml 추가 후보
  → 시험설계 리드 검토 후 yaml 반영
  → 다음 유사 제품 시험 시 TC_DESIGN 입력으로 자동 주입
```

#### 제품 유형 자동 분류

매뉴얼 텍스트에서 키워드를 분석해 제품 유형을 자동으로 추정하고,  
해당 유형의 자산만 선택해서 LLM에 주입한다.

| 제품 유형 ID | 설명 | 주요 키워드 |
|-------------|------|-----------|
| `BOARD_CMS` | 게시판/CMS | 게시판, 게시글, 댓글, 공지사항 |
| `USER_AUTH` | 회원/인증 | 회원, 로그인, 비밀번호, 권한 |
| `SHOPPING` | 쇼핑몰 | 주문, 결제, 장바구니, 재고 |
| `SEARCH` | 검색 | 검색, 필터, 정렬, 키워드 |
| `DASHBOARD` | 대시보드/분석 | 통계, 차트, 리포트 |
| `FORM_WORKFLOW` | 폼/워크플로우 | — |

---

### 다른 PC에서도 자산을 공유할 수 있는가?

**Phase 1(현재): 로컬 PC 한정**

`data/assets/` 폴더가 앱이 설치된 PC에만 존재한다.  
다른 PC에 설치된 AWT는 이 자산을 참조하지 못한다.

단, 결함 저장·승인 이벤트는 이미 **중앙 PostgreSQL**(`awt_asset_events` 테이블)에 기록된다.  
자산 파일 자체는 아직 중앙화되지 않았지만, 이력 추적 인프라는 갖춰져 있다.

```
현재 구조:
  [PC-A] data/assets/defect-catalog/  → 로컬에만 존재
  [PC-B] data/assets/defect-catalog/  → 로컬에만 존재 (PC-A 결과 모름)
  [중앙 DB] awt_asset_events           → "누가 언제 승인했는지" 이벤트만 기록
```

**당장 조직 내 공유가 필요하다면**

`data/assets/` 폴더를 내부 네트워크 드라이브에 두고 각 PC가 동일 경로를 참조하는 것이 가장 빠른 임시 방법이다.  
다만 동시 쓰기 충돌은 수동으로 관리해야 한다.

**Phase 2 계획: 중앙 자산 저장소 구축**

| 방식 | 장점 | 단점 |
|------|------|------|
| 자산도 PostgreSQL에 저장 | 앱 시작 시 자동 동기화 | DB 스키마 변경 필요, 오프라인 사용 불가 |
| 내부 Git 저장소에 `data/assets/` 관리 | 버전 관리 자연스러움, 승인 워크플로와 잘 맞음 | Git 인프라 필요, 앱에서 pull 자동화 필요 |
| 중앙 파일 서버 + 앱 시작 시 동기화 | 구현 가장 단순 | 충돌 해결 어려움 |

Phase 2에서 이 중 하나를 선택하면, 한 PC에서 발견·승인된 결함 패턴이  
전체 구성원의 AWT에 자동 배포되어 **조직 전체 TC 품질이 함께 올라가는 구조**가 된다.

| 항목 | Phase 1 (현재) | Phase 2 이후 |
|------|---------------|-------------|
| 결함 카탈로그 저장 위치 | 로컬 PC | 중앙 DB 또는 파일 서버 |
| 이벤트 로그(승인 이력) | 중앙 PostgreSQL | 중앙 PostgreSQL (동일) |
| 다른 PC 자산 참조 | ❌ 불가 | ✅ 가능 |
| 자산 동기화 | 없음 | 앱 시작 시 자동 pull |
| 패턴 누적 효과 | PC별 고립 | 전사 누적 → 많이 쓸수록 TC 품질 향상 |

**결론**: 시험 경험이 개인의 두뇌가 아닌 시스템에 저장되어  
누가 사용해도 일정 수준 이상의 결과를 보장한다.  
Phase 1은 로컬 PC 한정이나, 이벤트 로그 인프라가 이미 중앙화되어 있어  
Phase 2에서 자산 공유까지 자연스럽게 확장할 수 있다.
