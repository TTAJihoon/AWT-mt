# 02. 생산밀도 문제 — 왜 이게 본질인가

## 1. 외관 vs 실상

LLM에 충분한 컨텍스트(DOM + 이미지 + 매뉴얼 + ISO/IEC 25023 지침 + TC 샘플)를 주고 TC 생성을 요청하면 다음과 같은 출력 양상이 흔하다.

- TC 50~200개 생성
- 각 TC가 ISO/IEC 25023 품질특성 매핑 보유
- 시나리오·기대결과·assertion 모두 형식 정합
- 외관상 *풍부하고 체계적*

그러나 같은 프로젝트에서 *실제 결함 보고서* 와 대조하면:

| 항목 | 비율 |
|---|---|
| 결함에 직접 매핑되는 TC | 약 20~35% |
| 가치 없는 변종 (중복·사소·실현 불가) | 약 40~60% |
| 회사 정책에 어긋난 fictional TC | 약 10~20% |

**유효 TC 밀도가 사람 작성 대비 절반 이하** 가 되는 경우가 흔하다.

## 2. PoC-α 결과가 이 문제를 정확히 보여줌

```text
PoC-α 산출:
  41 TC 생성
  source_quote 직접 인용률: 58.5%
  INFERRED 비율: 41.5%
  PoC-γ 신규 결함 매핑률: 2/41 ≈ 4.9%
```

세 숫자가 다 *밀도 저하의 직접 지표*:

- **41.5% INFERRED**: LLM이 매뉴얼에 없는 영역을 *추론* 한 비율. 이 영역에서 *그럴듯하지만 회사 정책과 무관한* TC가 만들어질 위험.
- **58.5% source_quote**: 옳은 사상의 부분 작동. 나머지 41.5%가 위험 영역.
- **4.9% 신규 결함**: LLM 단독 성능. 베이스라인 추정 11% 미달.

이 세 숫자를 *임계 완화* 로 해결하는 건 *문제를 정상으로 인정* 하는 방향. 진짜 해법은 *추론 영역을 참조 영역으로 전환* — 즉 invariants 채널 + 결함 카탈로그.

## 3. 깊이의 세 정의

생산밀도 분석은 *깊이* 의 정의를 분리하지 않으면 헛돈다.

```text
정의 A. Cardinality depth
  같은 기능에 적용된 TC 개수

정의 B. Equivalence-class depth
  식별된 등가류(equivalence class) 수

정의 C. Information depth
  TC 1개가 검증하는 서로 독립인 invariant의 수
```

| 정의 | LLM 강약 | 사유 |
|---|---|---|
| A. Cardinality | 압도적 강함 | 인지부담 0, 끝까지 enumerate |
| B. Equivalence-class | 사람보다 강함 | 체크리스트형 사고 |
| C. Information | 사람보다 약함 | 도메인 invariant 부재 |

**A·B의 강점이 C의 약점을 가린다.** 그래서 *외관상 깊어 보이지만* 실효는 낮다.

AWT-claude의 7기법(happy_path / equivalence / boundary / negative_basic / negative_deep / state_transition / cross_feature)은 *기법 분포 강제* 로 B를 부분 보강한다. 하지만 *각 기법 안에서의 깊이* 와 *C 영역* 은 자산이 없으면 보강 불가.

## 4. 외관상 깊어 보이는 세 패턴

### 4.1 Cosmetic Depth (위장된 깊이)

같은 결함을 잡는 TC를 *다른 모양으로 재포장* 한 것.

```text
LLM 출력 예
TC-A: "정상 등록 후 즉시 로그인 시도 시 정상 동작 확인"
TC-B: "회원가입 흐름 end-to-end 검증"
TC-C: "신규 사용자 등록 후 로그인 기능 연동 확인"

→ 세 TC가 잡는 결함 집합 거의 동일
→ TC를 단순화하면 한 개로 묶임
→ unique_value = 0 인 2개의 cosmetic TC
```

V4(기법 분포)로는 이걸 못 잡는다. *invariant 집합 Jaccard* 같은 별도 검증이 필요. → `03-seven-design-assets.md` §2.7.

### 4.2 Spec Hallucination (지어낸 사양)

도메인 매뉴얼에 명시되지 않은 *그럴듯한* 정책을 LLM이 가정하고 TC를 작성하는 현상.

```text
LLM 출력 예 (회원가입)
TC: "동일 IP에서 5분 내 3회 이상 계정 등록 시 자동 차단되는지 확인"

문제:
- 회사 정책에는 그런 차단 규칙이 없음
- LLM이 "있을 법한 보안 정책"을 시나리오로 만듦
- 검수자가 정책 문서 확인 안 하면 "결함처럼" 보고됨
```

source_quote 강제로 부분 차단 가능. 하지만 LLM은 *quote 형식을 흉내내는 출력* 도 만들 수 있어서 100% 차단 안 됨. 진짜 해법은 *invariants 채널* — LLM이 *quote할 수 있는 정책 문서가 존재* 하게 만들기.

### 4.3 Equivalence-class Inflation (등가류 부풀리기)

같은 등가류 안에서 표면값만 바꾼 변종을 다수 생성.

```text
LLM 출력 예 (이메일 검증)
TC-1: 이메일 "abc"        → 형식 오류
TC-2: 이메일 "xyz"        → 형식 오류
TC-3: 이메일 "12345"      → 형식 오류
TC-4: 이메일 "abc.def"    → 형식 오류
TC-5: 이메일 "abc@"       → 부분 형식 결손
TC-6: 이메일 "@def"       → 부분 형식 결손

→ TC-1~3은 같은 등가류 (도메인 없는 짧은 문자열)
→ TC-4~6은 부분적 형식 결손, 서로 다른 등가류
→ 6개 중 실효 3개, 나머지 3개는 inflation
```

V4(분포)는 *기법* 만 보지 *등가류* 는 안 본다. → `03-seven-design-assets.md` §2.1 등가류 템플릿 라이브러리.

## 5. 회원가입 사례 — 사람 5 vs LLM 25 깊이 분석

### 5.1 사람의 TC 5개 (시니어)

```text
1. 정상 등록 (필수값만)
2. 필수값 누락 → 검증 메시지
3. 중복 이메일 → 거부
4. 비밀번호 정책 위반 → 거부
5. 비활성화 계정 재등록 시도 → 정책에 따른 처리
```

5개 모두 *서로 다른 결함을 잡음*. 정보 깊이 명확.

### 5.2 LLM의 TC 25개 (충분한 컨텍스트 주입 시)

```text
[정상 등록 변종] 2개
[필수값] 4개
[이메일 형식] 6개
[이메일 정규화] 3개
[비밀번호] 4개
[중복] 3개
[보안성 ← spec hallucination 위험] 2개
[상태 처리] 1개
```

### 5.3 25개의 실효 평가

| 분류 | 개수 | 비고 |
|---|---:|---|
| 사람 5개와 *직접 대응* | 5 | 등가 깊이 |
| 추가 *유효* 변종 (사람이 빠뜨린 등가류) | 6~8 | LLM의 진짜 우위 |
| 등가류 inflation | 4~6 | 가치 없음 |
| Cosmetic 중복 | 2~3 | 가치 없음 |
| Spec hallucination | 2~4 | 부정적 가치 |
| 도메인 부적합 | 2~3 | 가치 없음 |

**유효 TC = 11~13개. 25개 중 절반 미만.**

자동화 우위는 *11~13 vs 5* = 약 2배. 외관 *25 vs 5* = 5배가 아니다.

## 6. AWT-claude의 41 TC를 이 프레임으로 재해석

```text
41 TC 생성 (PoC-α)
  ≈ 정상 변종 5
  ≈ 등가류 변종 12
  ≈ 음성 케이스 8
  ≈ 상태 전이 6
  ≈ cross_feature 4
  ≈ 기타 6

자산 미적용 LLM의 전형적 분포에서:
  - 유효 (결함 잠재력 있는) TC ≈ 18~22개
  - 등가류 inflation + cosmetic ≈ 12~15개
  - spec hallucination 의심 ≈ 4~6개

PoC-γ에서 9 fail / 32 pass, 신규 결함 2개
→ 18~22개 유효 TC 중 2개에서 신규 결함 발견 = 약 10%
→ 32 pass TC 중 *진짜 검증* 한 게 얼마인지 알 수 없음 (cosmetic 다수 가능)
```

이게 *41 TC를 생성했지만 신규 결함은 2개* 의 구조적 설명. 단순한 LLM 한계가 아니라 *자산 부재 + cosmetic 통과* 의 복합 결과.

## 7. 해결의 본질 — LLM 튜닝이 아니라 입력 자산

생산밀도 저하는 *LLM의 결함* 이 아니라 **입력으로 제공된 자산의 결함** 이다.

```text
DOM + 이미지 + TC 샘플 + 매뉴얼 + ISO 25023 → 일반 정보, 보편적
                                                ↓
                                        보편적 TC 출력 (밀도 낮음)

[추가 입력이 필요한 영역]
+ TestPattern 라이브러리         → 등가류 분류, 우선순위, skip 조건
+ domain-invariants.yaml        → 회사·정책 특수성
+ cross-screen-invariants.yaml  → 화면 간 관계
+ past-defect-patterns RAG      → 결함 학습
                                                ↓
                                       *밀도 있는* TC 출력
```

이 추가 입력 자산이 *AWT-claude에 아직 없는* 부분. 본 제안의 핵심 권고는 이걸 만들자는 것.

## 8. AWT-claude 적용 시 기대 효과

자산 도입 시점별 결함 매핑률 추정 (PoC-α 환경 → 진짜 제품 시험 환경 보정 후):

| 시점 | 자산 상태 | 결함 매핑률 |
|---|---|---|
| 현재 (PoC) | 자산 없음 | 8~14% (목업 보정 후) |
| 1개월 후 | 결함 카탈로그 + invariants + selector 점수 추가 | 12~15% |
| 6개월 후 (Phase 1 종료) | 1개 제품유형 TestPattern 30~50개 | 14~17% |
| 1년 후 (Phase 2 종료) | 3~5개 제품유형 + 결함 카탈로그 300~500건 | 15~18% |
| 2년 후 (Phase 3 + RAG) | 결함 카탈로그 800~1500건 임베딩 | 18~21% |

**천장은 약 21%.** 사람 22~25%에 *완전 도달은 영원히 못 함* (시니어 베스트의 암묵지 한계). 하지만 *사람 평균* 에는 도달 가능.
